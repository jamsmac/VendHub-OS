## Key decisions and rationale

### P1-I deferred: BullMQ analytics queue not needed

**Decision:** Do not move `analytics.listener.ts` to BullMQ.

**Why:** The listener uses incremental upserts (`DailyStats` row update per event), not heavy rebuilds. Each handler does a single `findOne` + `save` — sub-10ms per call. Adding BullMQ would introduce queue round-trip latency (~50-200ms) for zero throughput gain.

**Rule 10 criteria for BullMQ offload:**

1. Handler touches >2 tables — **no** (touches 1: `DailyStats`)
2. Handler updates >3 rows per event — **no** (updates exactly 1)
3. Handler makes external HTTP calls — **no**
4. Handler takes >100ms — **no** (single upsert)

**When to revisit:** If the analytics listener grows to do full aggregation rebuilds (joining transactions + collections + machines), or if burst event volume causes DB connection pool exhaustion. Track via: Prometheus `event_handler_duration_seconds{handler="analytics.listener"}`.

### /dashboard/promotions and /dashboard/dashboard/\* deferred

**Decision:** Keep both pages as-is. Only add 308 redirects for the 4 simple duplicates.

**Why:**

- `/dashboard/promotions` = 2037 LOC across 9 components (wizard, analytics, A/B tests, coupons, KPI cards). This is real product functionality, not a duplicate to delete. Consolidating it into `/dashboard/loyalty/promo-codes` is a product decision — the two pages serve different use cases (promotions = campaign management, promo-codes = code lifecycle).
- `/dashboard/dashboard/*` = 1266 LOC across 10 tab components (sales, activity, machines, hourly chart, KPI cards, alerts, quick actions). This is the main dashboard's sub-navigation. Moving it up one level is a routing refactor that touches layout.tsx, breadcrumbs, and sidebar — mechanical but needs coordinated testing.

**When to revisit:** Sprint B, after product sign-off on the canonical promo UX.

### alerts.triggered (plural) as canonical event name

All alert emits now use `alerts.triggered` (plural). The prior state had:

- `alerts.service.ts:227` → `alerts.triggered` (plural, correct)
- `alert-evaluator.service.ts:304` → `alert.triggered` (singular, wrong)

Listeners were subscribed to the plural form, so evaluator-triggered alerts were silently dropped. This was a live bug — alerts from the cron evaluator never reached the notification system.

**Convention:** All event names in the alerts module use the plural entity prefix: `alerts.triggered`, `alerts.resolved`, `alerts.acknowledged`. This matches the rest of the codebase (`complaint.created`, `referral.activated`) except that alerts uses plural because the evaluator fires batches.

### P0-B: ephemeral key fingerprint logging

The new `getEncryptionKey()` logs an 8-char SHA-256 fingerprint of the ephemeral key on first use:

```
[TwoFactorService] Using ephemeral dev TOTP key (fingerprint=a1b2c3d4).
  TOTPs encrypted with this key will be unreadable after restart.
```

This lets developers correlate "my 2FA stopped working after restart" with "the key rotated." Without it, the symptom is a cryptic `Unsupported state or unable to authenticate data` from AES-GCM, which looks like a bug rather than expected behavior.

### P0-C: multi-use feedback tokens

Feedback tokens are multi-use (no `token_used_at` column, no migration). Rationale:

- Attack surface is low: submitting duplicate feedback for a complaint you legitimately received a token for is not a meaningful security risk
- A DB migration + entity change + service branch for single-use is cost-disproportionate
- "Latest wins" is natural for the `satisfactionRating` / `satisfactionFeedback` fields — they're scalars, not append-only

If feedback spam becomes a problem, add a rate limit per token (1 submission/hour) — cheaper than single-use.
