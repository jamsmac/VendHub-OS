# ADR-002: Модель ролей — 4 staff + Client + конструктор галочек

**Status:** Accepted (реализация после стабилизации пилота)
**Date:** 2026-04-29
**Decision-makers:** Jamshid Sadikov (Owner)

---

## Context

В проекте 7 жёстких ролей: OWNER, ADMIN, MANAGER, ACCOUNTANT, WAREHOUSE, OPERATOR, VIEWER. Они описаны в `packages/shared/src/types/user.types.ts` и используются через декораторы `@Roles()` в 1067 эндпоинтах.

При обсуждении выявились ограничения:

1. У разных франчайзи может быть разная структура — 7 ролей не покроют все случаи без custom-roles
2. Бухгалтер, кладовщик, оператор — это часто **набор прав**, а не должность. Один человек может совмещать.
3. Admin сейчас имеет доступ ко всем организациям, что несовместимо с франчайзи-моделью
4. Owner может быть любым с ролью OWNER — нет гарантии, что owner единственный
5. Клиенты (покупатели у автоматов) живут в той же таблице `users` — это техдолг

## Decision

Переходим на модель **4 staff-ролей + 1 client-роль + per-user конструктор прав**.

### Структура

| Роль         | Описание                                                                                                     |
| ------------ | ------------------------------------------------------------------------------------------------------------ |
| **Owner**    | Единственный пользователь — Jamshid. Владелец платформы, доступ ко всем организациям.                        |
| **Admin**    | Глава одной организации. По дефолту — все права своей орг + обе мета-галочки создания (Manager, Employee).   |
| **Manager**  | Руководитель команды или специалист. Создаёт Employee только если Admin включил `can_create_employees`.      |
| **Employee** | Сотрудник без подчинённых. Шаблоны: Оператор / Кладовщик / Бухгалтер / Наблюдатель.                          |
| **Client**   | Покупатель. Отдельная таблица `clients`, отдельный JWT audience `vendhub-clients`, не пересекается со staff. |

### Жёсткие правила

1. **Owner — singleton.** Проверка по конкретному user_id или явное правило в коде.
2. **Admin scoped к одной орг** через `user.organizationId`.
3. **Делегирование «только из своих»** — `assertCanGrant(grantorId, permission)` отбивает попытку выдать чего нет.
4. **Мета-галочки** `can_create_managers` / `can_create_employees` — отдельные permissions, не свойства роли.
5. **Staff и Client изолированы** — разные таблицы, разные эндпоинты логина (`/auth/login` vs `/auth/client/telegram`), разный JWT audience.
6. **Управление в одной панели** — единая UI-страница, не разрозненные экраны «Менеджеры», «Операторы».

### 8 категорий галочек-конструктора

Поле, Склад, Деньги, Каталог, Лояльность, Машины, Отчёты, Админка (только Admin).

## Rationale

### Почему 4 staff а не 7

- **«Бухгалтер» = набор прав на категорию «Деньги»**, не должность. Тот же набор может иметь Manager-финансист или Employee.
- **«Кладовщик» = набор прав на «Склад»**.
- **«Оператор» = набор прав на «Поле»**.
- **«Наблюдатель» = всё read-only**.

Жёсткие роли заменяются **именованными пресетами** в UI, которые проставляют галочки. Это даёт ту же простоту использования + гибкость.

### Почему Client отдельной ролью, а не Employee

- Клиенты регистрируются сами (через Telegram Mini App или phone+OTP), без приглашения от admin
- У них свой кабинет с другими данными (бонусы, история покупок, рефералы)
- JWT audience должен быть разный — клиентский токен не должен проходить на staff-эндпоинты, даже теоретически
- Один клиент может покупать у нескольких организаций (франчайзи) — нужна связь many-to-many через `client_organizations`

### Почему один Owner

- Owner = владелец **платформы**, не одной организации. По определению один.
- Без ограничения легко создать «второго owner» по ошибке через PATCH /users/:id с role=OWNER → privilege escalation.

## Consequences

### Positive

- Гибкость для франчайзи (каждый Admin — на свою орг, свой набор прав)
- Меньше жёсткости — два менеджера могут иметь разные права
- Чёткое разделение Staff/Client — снижение PII-рисков
- Owner physically singleton — нельзя случайно создать ещё одного

### Negative

- Большая миграция — 8 этапов, ~4-6 недель работы
- Все 1067 декораторов `@Roles` со временем мигрируют на `@Permissions`
- Существующие пользователи нужно мигрировать в новые шаблоны

### Mitigation

- **Не делать перед пилотом.** Текущая модель работает, протестирована. Реализация после стабилизации.
- **Параллельные декораторы** — `@Roles` и `@Permissions` сосуществуют до завершения миграции. Никакого big-bang.
- **Скрипт миграции** проверяет «доступы до и после миграции совпадают» для каждого пользователя.

## Implementation phases

8 этапов в `role-redesign-decision.md`. Группированы в 5 спринтов в `02-phase3-execution-plan.html`.

1. PermissionsGuard и интеграция с существующим RBAC
2. Правило assertCanGrant
3. Мета-галочки can*create*\*
4. Сокращение enum UserRole с 7 до 4
5. Admin привязан к одной организации
6. Разделение users и clients
7. UI единой панели управления
8. Миграция декораторов с @Roles на @Permissions

## What enables this decision

В коде уже есть значительный RBAC-фундамент, который не используется массово:

- Модуль `rbac` с динамическими ролями и permissions (4 таблицы)
- 12 эндпоинтов в RbacController
- Permission entity формата `resource:action`
- Роли с уровнями (`level` поле)
- Системные роли (`isSystem=true`) защищены

**Не строим с нуля — доинтегрируем существующее.**
