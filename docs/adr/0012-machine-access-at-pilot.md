# ADR-004: Resource-level доступ к машинам активируется на пилоте

**Status:** Accepted
**Date:** 2026-04-29
**Decision-makers:** Jamshid Sadikov (Owner)

---

## Context

В коде уже существует модуль `apps/api/src/modules/machine-access` с полной инфраструктурой resource-level доступа к автоматам:

- Таблица `machine_access(organizationId, machineId, userId, role, granted_at)`
- Уникальный индекс `(machineId, userId)` с `WHERE deleted_at IS NULL`
- Enum `MachineAccessRole` с 5 значениями: FULL / REFILL / COLLECTION / MAINTENANCE / VIEW
- 13 эндпоинтов: создание, отзыв, выборки, шаблоны массового назначения

Однако этот модуль **не используется массово** в основных машинных эндпоинтах. Менеджер видит все машины своей организации, не только закреплённые за ним.

Изначально resource-level права были запланированы в Фазе 4 (постмиграционные улучшения). Во время обсуждения роль-редизайна стало ясно, что это **существенно дешевле в реализации, чем казалось** (фундамент готов).

## Decision

**Включить resource-level доступ к машинам на пилоте**, в Фазе 1 «Перед запуском пилота», не откладывая до Фазы 4.

## Rationale

### Почему сейчас, а не позже

1. **Фундамент готов.** Не нужно строить с нуля. Нужно только подключить существующий машинный доступ к фильтрации и добавить UI назначения.

2. **Решает реальную потребность пилота.** Если Jamshid захочет нанять регионального менеджера и дать ему 5 машин в районе Бостандыка — без machine-access он автоматически получит доступ ко всем 23.

3. **Это сильное конкурентное преимущество.** Большинство SaaS не поддерживают resource-level из коробки. Для франчайзи-модели это критично.

4. **Низкий риск.** Если фильтр не подключён — менеджер видит всё (как раньше). Если подключён — видит только своё. Бэкап-сценарий — всегда возможность отозвать в admin-панели.

### Что нужно сделать

- **Подключить фильтрацию** в основные `machines.controller` методы — выборки фильтруются по machine_access для не-Owner/Admin ролей
- **UI назначения** в admin-панели — экран «Закрепить пользователя за машинами»
- **Default-логика** — для Manager при создании автоматически закрепляются все машины орг (пока админ явно не уточнит)

## Consequences

### Positive

- Готовность к франчайзи (региональные менеджеры)
- Тонкий контроль доступа уже на пилоте
- Демонстрация продвинутой архитектуры инвесторам/клиентам
- Reuse существующего кода

### Negative

- Дополнительные 1-2 дня работы перед запуском
- UI назначения нужно сделать минимальный, без bells&whistles

### Mitigation

- На пилоте Owner = Jamshid имеет full bypass, поэтому даже если фильтр сломается, он всё равно видит все машины. Не критический путь.
- Manager тестируется по сценарию из smoke-test (см. `LAUNCH_CHECKLIST.md` пункт 7.10).

## Implementation tasks

(Войдут в Фазу 1 запуска)

1. ✅ Audit модуля machine-access (выполнен в `MASTER.html` секция G)
2. ⏳ Подключить `MachineAccessService.canAccess(userId, machineId)` в `MachinesService.findAll`, `findById`, `update`
3. ⏳ Сделать страницу `/dashboard/users/:id/machine-access` в админке
4. ⏳ Default-логика при создании Manager — назначить все машины орг с типом FULL
5. ⏳ Smoke-тест: создать тестового manager, привязать к 2 машинам, проверить что видит только их

## What this ADR does NOT promise

- **Не custom permissions per machine** — тип доступа (FULL/REFILL/etc) на уровне записи, не отдельные действия
- **Не cross-organization machine sharing** — машина принадлежит одной организации, не нескольким
- **Не наследование от региона** — если хотите «все машины Бостандыкского района», нужно явно перечислить
