Самая «бухгалтерская» группа. Каждое движение товара (приёмка, перемещение между складами, пополнение машины, продажа) фиксируется как event, потом материализуется в balances. Это event-sourced архитектура из Sprint G1.

⌘Архитектура event-sourced

Sprint G1 ввёл новую модель: вместо прямого UPDATE остатков в таблице — event-sourced подход.

1

StockMovement event

Каждое физическое движение (приёмка, выдача, пополнение, продажа) пишется как append-only event с типом, источником, целью, productId, quantity, ts.

2

InventoryBalance — материализованный остаток

Listener на StockMovement события пересчитывает balance per (productId, locationId, batchId).

3

Опт-снапшоты

Раз в сутки делается snapshot балансов — для быстрого rollback и аудита.

4

QuantitySyncService

При transaction.created event декрементирует MachineSlot.currentQuantity. Это ключевой связник между Payments и Inventory.

Критический риск: QuantitySyncService должен быть верифицирован на проде. P2.1 в ROADMAP — без этого предсказания пополнения работают на устаревших данных.

▣Детали по модулям

inventory · главный модуль остатков13 файлов · 14 эндпоинтов

Назначение: read-API для остатков на всех уровнях иерархии (склад → ёмкость → продукт).

Эндпоинты: GET /inventory (overview), /by-product, /by-location, /by-machine, /low-stock, /expiring, /movements (history), /balance/:id, POST /adjust (ручная корректировка), POST /transfer.

RBAC: Owner/Admin/Manager — все 14, Warehouse — все 14 (его основная зона), Operator — 0 (через machine-access видит свои машины).

Code smell: бухгалтер закрыт от inventory — 0 доступа. Может быть правильно (склад не его дело), но стоит подтвердить с реальным бухгалтером.

warehouse · склады и операции13 файлов · 25 эндпоинтов

Назначение: CRUD складов, приёмка от поставщиков, исходящие перемещения, инвентаризация.

Самый «нагруженный» эндпоинтами модуль группы. Покрывает всю операционную работу склада.

Связь: contractors (поставщики), purchases (закупки), inventory (остатки), batch-movements (партии).

storage · локации хранения16 файлов · 14 эндпоинтов

Назначение: разделение склада на зоны (стеллажи, полки, холодильники). У товара может быть конкретная storage-location.

RBAC: Operator имеет 9/14 — через machine-access работает с конкретными storage при пополнении.

containers · паллеты, ящики8 файлов · 11 эндпоинтов

Назначение: отслеживание физических контейнеров с товаром. Один контейнер может содержать много продуктов.

RBAC: Warehouse 8/11, Operator 7/11 — оба много работают с контейнерами при пополнении автоматов.

opening-balances · начальные остатки6 файлов · 9 эндпоинтов

Назначение: загрузка начальных остатков при первом запуске системы или при импорте из старой системы.

Каждый opening_balance — это виртуальный StockMovement в нулевую дату. После него все остальные движения накладываются последовательно.

inventory-reconciliation · сверка после инвентаризации8 файлов · 6 эндпоинтов

Назначение: когда оператор делает физическую инвентаризацию автомата и считает реальный остаток — этот модуль фиксирует расхождение между фактом и учётом.

Расхождение → корректировка через StockMovement с типом ADJUSTMENT + комментарий. Audit-лог обязателен.

stock-movements · event listener4 файла · 0 HTTP

Internal сервис без контроллера. Слушает transaction.created, warehouse.received, machine.refilled и пишет StockMovement records.

Это ядро event-sourced архитектуры. Если упадёт — балансы перестанут обновляться.

batch-movements · партии товара5 файлов · 3 эндпоинта

Назначение: отслеживание партий по производственной дате и сроку годности. Когда продаётся товар — списывается из самой ранней партии (FIFO).

Связь с алертами: алерт EXPIRING_SOON срабатывает за 7 дней до истечения срока партии.

inventory-dashboard · сводный экран3 файла · 1 эндпоинт

Один эндпоинт-агрегатор GET /inventory-dashboard, возвращающий всё для главного экрана: топ-движения, low-stock алерты, expiring soon, общая стоимость склада.
