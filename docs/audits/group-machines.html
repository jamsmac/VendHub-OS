▣Детали по модулям

machines · сами автоматы23 файла · 44 эндпоинта · 2 теста

Назначение: CRUD автоматов, паспорта, статусы, слоты с товаром, шаблоны, телеметрия, события.

Entity главные

  • Machine — id, серийник, type (COFFEE / SNACK / etc), status (ACTIVE / OFFLINE / MAINTENANCE), organizationId, locationId, координаты
  • MachineSlot — позиция в автомате (slot 1, slot 2...), productId, currentQuantity, maxCapacity, sellingPrice
  • MachineTemplate — типовые конфигурации автоматов
  • MachineEvent — события: открыт корпус, ошибка, инкассация, пополнение

Топ-эндпоинты (из 44)

  • GET /machines — список с фильтрами по статусу / location / типу
  • GET /machines/:id — детальная карточка с слотами и событиями
  • PATCH /machines/:id/status — изменить статус (Operator может через machine-access)
  • GET /machines/:id/slots — позиции с остатками
  • POST /machines/:id/slots/:slotId/refill — пополнение оператором
  • GET /machines/:id/events — журнал событий

RBAC: 50 декораторов в матрице — Owner/Admin/Manager полный доступ, Operator 23/50 (свои машины через machine-access).

Связи: products (что продаётся), inventory (склад товара), payments (транзакции), routes (маршруты пополнения), predictive-refill (прогноз), websocket (realtime).

equipment · оборудование автоматов16 файлов · 26 эндпоинтов · 4 контроллера

Структура: 4 sub-controller'а: equipment (общий), hopper-type, spare-part, washing-schedule.

Что хранит: компоненты автоматов (hopper для зёрен, для молока, для воды), запчасти со складом, расписания мойки.

Связь с maintenance: заявка на ремонт ссылается на конкретный component через FK.

Тестов: 8 — самое плотное покрытие в группе.

websocket · realtime через 4 gateway8 файлов · нет HTTP endpoints

WebSocket Gateway'и работают по handshake-аутентификации через JWT. Не контроллеры в классическом смысле.

4 gateway

  • base.gateway.ts — общая логика (auth, organizationScope)
  • machine-events.gateway.ts — статусы машин в realtime
  • order-events.gateway.ts — заказы в realtime для дашборда
  • notification.gateway.ts — push-уведомления операторам

Tenant scoping: при подключении клиент попадает в namespace `org:`. События не утекают между организациями.

Используется в web админке: hook use-realtime-updates.ts, lazy-import socket.io-client.

Risk: persistent connections — на Railway держится через TCP keepalive. Vercel-стиль serverless их не поддерживает (это одна из причин Решения 1 — всё на Railway).

alerts · правила и срабатывания9 файлов · 13 эндпоинтов

Назначение: alert_rules (когда срабатывать) и alert_events (когда сработало). Поддерживает SMS, email, push, webhook.

AlertMetric enum: LOW_STOCK, MACHINE_OFFLINE, FAILED_PAYMENT, PREDICTED_STOCKOUT и др.

Suppression window: ≥24h per (machine, product) — предотвращает alert fatigue.

Action item: seed для PREDICTED_STOCKOUT alert rule per active org нужно сделать перед включением predictive-refill (см. LAUNCH_CHECKLIST шаг 5).

predictive-refill · прогноз пополнения автоматов11 файлов · 4 эндпоинта · 4 теста

Назначение: главная «умная» фича. Каждые сутки EWMA (exponentially weighted moving average) считает скорость потребления каждого слота → forecasting daysOfSupply → recommendations какие машины пополнять первыми.

3 REST эндпоинта

  • GET /predictive-refill/forecast/:machineId — days-of-supply per slot
  • GET /predictive-refill/recommendations?action=... — sorted recommendations
  • POST /predictive-refill/recommendations/:id/mark-acted — audit trail когда оператор пополнил

Состояние

  • Backend engine API-complete (Sprint E Phase 2)
  • Cron не настроен — recommendations обновляются только при ручном HTTP запросе (Phase 3 ROADMAP)
  • Web UI на /dashboard/predictive-refill — данные подгружаются с этих эндпоинтов
  • PREDICTED_STOCKOUT alerts не fire даже если daysOfSupply < 2 (Phase 3 ROADMAP)

Зависимость от данных

Работает только если MachineSlot.currentQuantity декрементируется при каждой продаже. Это P2.1 в ROADMAP — нужно верифицировать на проде через тестовую покупку.

monitoring · system metrics с защитой токеном5 файлов · 2 эндпоинта

@Public + MetricsKeyGuard. Эндпоинты публичны для bypass JWT, но проверяют METRICS_API_KEY.

Без токена в env эти эндпоинты становятся открытыми. Action item — выставить в Railway (см. LAUNCH_CHECKLIST шаг 3).

Append-only log изменений конфигурации слотов автомата. Когда был добавлен товар, когда заменён, кем.

geo · гео-координаты и зоны7 файлов · 8 эндпоинтов

PostGIS extension для геопоиска. Все 7 ролей могут читать (geo — справочник).

Use case: «найти ближайшие 5 автоматов к моей точке» — для Telegram Mini App клиента.

locations · где стоят автоматы17 файлов · 6 эндпоинтов

Что: бизнес-центры, ВУЗы, ТРЦ, метро. Иерархия: Location → Building → Floor → Spot.

Один Location может иметь много машин. Через Location можно сгруппировать выручку и аналитику.

✓Готовность группы

🟢 Группа технически готова к запуску пилота. Из 10 модулей только 2 требуют env-настройки или seed (alerts, monitoring), оба входят в Фазу 1 LAUNCH_CHECKLIST.

Что включить параллельно с запуском:

  • Resource-level фильтрация в machines.controller через machine-access (ADR-004)
  • Seed PREDICTED_STOCKOUT alert rules
  • Smoke-тест предсказаний — открыть /dashboard/predictive-refill, увидеть данные

Что после пилота:

  • Cron processor для predictive-refill (Sprint E Phase 3)
  • Wire-up PREDICTED_STOCKOUT в alerts pipeline
  • Integration test суточного цикла «продажа → расход → прогноз → рекомендация»
