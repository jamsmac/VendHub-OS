health · health checks9 файлов · 5 эндпоинтов

Назначение: liveness/readiness probes для Railway, Kubernetes, мониторинга.

Эндпоинты (все @Public)

  • GET /health — basic «жив ли сервис», возвращает {status: ok}
  • GET /health/live — liveness probe для k8s, проверка memory heap
  • GET /health/ready — readiness probe, проверка БД и Redis (вернёт 503 если что-то не работает)
  • GET /health/detailed — полная диагностика всех компонентов
  • GET /health/version — версия билда

Это то что railway.toml использует для healthcheck. Если /health/ready возвращает 503 — Railway не маршрутизирует трафик на этот инстанс.

monitoring · с защитой токеном5 файлов · 2 эндпоинта

@Public + MetricsKeyGuard. Защита через METRICS_API_KEY в env.

Назначение: для внешних мониторинг-систем (Grafana, custom dashboards) которые опрашивают метрики.

Risk: если METRICS_API_KEY не выставлен — эндпоинт становится открытым. Action item — обязательно прописать в Railway env.

metrics · Prometheus endpoint3 файла · 1 эндпоинт

@Public + проверка METRICS_TOKEN. Если токен задан — обязателен в query ?token=... или header x-metrics-token.

Назначение: для Prometheus / Grafana Agent — scraping метрик в формате Prometheus text/plain.

Что метрики покрывают: http_requests_total, request_duration, payment_success_rate, login_success_rate, db_pool_active, redis_connections и др.

entity-events · журнал событий entities6 файлов · 4 эндпоинта

Назначение: append-only лог изменений entity (отдельно от audit log). Используется для дебага и replay.

Отличие от audit:

  • audit_log — high-level действия пользователя (logged in, exported report)
  • entity_events — low-level изменения сущностей (Machine#X статус изменился с ACTIVE на MAINTENANCE)

RBAC: Owner 4/4, Admin 4/4, Manager 4/4. Operator/Warehouse — 0 (это для дебага).

bull-board · UI BullMQ очередей3 файла · 1 эндпоинт

Назначение: графический интерфейс для просмотра BullMQ-очередей (`fiscal`, `machine-writeoff`, web-push, email).

URL: /admin/queues — только Owner.

Use case: мониторинг очередей в проде — сколько в backlog, сколько failed, retry.

Когда полезно: Soliq.uz упал — failed fiscal jobs накапливаются в bull-board, видно сразу.

✓Готовность группы

🟢 Готова к запуску при правильной настройке env. Группа маленькая, но критичная — без неё мониторинг и observability не работают.

Что обязательно перед запуском:

  • METRICS_API_KEY и METRICS_TOKEN в Railway env
  • Настроить Prometheus / Grafana Agent на скрапинг /metrics с токеном
  • Регулярно проверять /admin/queues — особенно после первой пары дней
  • Настроить алерт «healthcheck /health/ready возвращает 503 более 5 минут»
