routes · маршруты с автогенерацией19 файлов · 33 эндпоинта

Назначение: построение маршрутов оператора на день. Каждое утро в 5:00 авто-генерация маршрутов на основе recommendations от predictive-refill.

Sprint H — MergeTripsIntoRoutes

Раньше были отдельные сущности Trip и Route. В миграции 1775700000000 они объединены в Route — упрощение модели.

Структура

  • Route — id, operatorId, date, status, machines[], plannedDuration
  • RouteStop — точка маршрута, конкретный machineId, action (REFILL / COLLECT / MAINTAIN), expectedArrival, actualArrival

Auto-route logic

Из predictive-refill берутся машины с REFILL_NOW recommendation, группируются по геолокации, строится оптимальный маршрут (TSP-приближение).

Кнопка «Авто-маршрут» в web — Manager одной кнопкой создаёт маршруты на следующий день для всех операторов.

RBAC: Operator 15/33 (свои маршруты + действия). Manager 32/33. Admin 33/33.

tasks · задачи в kanban-стиле12 файлов · 27 эндпоинтов

Структура: Task с статусами TODO / IN_PROGRESS / DONE / CANCELLED. Привязан к assignee (operatorId), может быть привязан к machineId.

UI: KanbanBoard в web для Manager + список «Мои задачи» в mobile staff app.

RBAC: Operator 18/27 — может видеть свои, обновлять статус, добавлять комментарии. Manager 26/27 — управление и назначение.

maintenance · ТО и ремонт8 файлов · 26 эндпоинтов

Sprint H недавно: добавлена связь maintenance request с конкретным component (через FK на equipment.spare_part и equipment.hopper).

Web-push на manager + assigned tech при SLA breach — недавняя фича.

Quick mark-completed — оператор одной кнопкой закрывает работу без полного workflow.

Связь: equipment (что чинить), incidents (откуда возникла потребность), tasks (как работа).

work-logs · журнал работ оператора7 файлов · 23 эндпоинта

Назначение: каждое физическое действие оператора у автомата фиксируется WorkLog с timestamp, типом действия, machineId, locationId.

QR-фиксация обязательна: оператор сканирует QR на автомате чтобы система зафиксировала «он действительно был на месте».

Use case: KPI оператора, сверка с маршрутом, доказательство выполнения.

collections · инкассация6 файлов · 13 эндпоинтов

Назначение: выемка наличных из автоматов. Оператор фиксирует факт инкассации, сумму, делает фото купюр.

Связь с machine-access: только пользователи с типом доступа COLLECTION или FULL могут проводить инкассацию.

Связь с reconciliation: сверка факта инкассации (что взяли из машины) с финансовыми отчётами (что должно быть).

incidents · инциденты с автоматами8 файлов · 10 эндпоинтов

Назначение: «автомат не работает», «застрял стакан», «нет молока». Workflow: диагностика, назначение оператора, SLA, разрешение.

Связь: может породить maintenance запрос. Связан с complaints (если жалоба клиента стала причиной инцидента).

SLA timer в UI — Manager видит просрочки.

reconciliation · сверка6 файлов · 8 эндпоинтов

Назначение: ежедневная сверка cash-инкассации с банковскими выписками и платёжными отчётами.

Расхождения: создаётся issue для Accountant — разобраться почему касса не сходится с отчётами.

Use case: оператор обнаружил что в его авто закончились запасные стаканчики — создаёт material-request → Manager одобряет → Warehouse выдаёт со склада.
