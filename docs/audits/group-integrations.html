▣Детали ключевых модулей

telegram-bot · самый сложный модуль34 файла · 2 controllers · 13 эндпоинтов · много services

Назначение: весь Telegram-функционал — клиентский бот, мини-приложение, webhook от Telegram.

Ключевые сервисы (из 34 файлов)

  • telegram-bot.service.ts — главный сервис
  • telegram-customer-bot.service.ts — логика для клиентов
  • bot-order.service.ts — обработка заказов внутри бота
  • access-request.entity.ts — запросы доступа из бота

2 controller'а

  • telegram-bot.controller.ts (11) — admin endpoints для управления ботом
  • telegram-webhook.controller.ts (2) — webhook от Telegram (Public, защищён secret)

Webhook secret: TELEGRAM_WEBHOOK_SECRET — генерится при setup, проверяется на каждом запросе

Связь: auth (Telegram login), orders (заказы из бота), notifications (push клиентам через бот).

notifications · hub всех уведомлений18 файлов · 29 эндпоинтов

Назначение: единая точка входа для отправки уведомлений по любому каналу (email, SMS, push, in-app, Telegram).

Каналы: email (через email модуль), SMS (через sms модуль / Eskiz.uz), web-push, FCM, Telegram bot message, in-app banner.

Шаблоны: хранятся отдельной сущностью NotificationTemplate с placeholders.

RBAC: 12/29 для всех ролей — настраивают кому что приходит.

integrations · общий hub15 файлов · 22 эндпоинта

Назначение: управление API-ключами и статусами всех внешних интеграций. Admin может включить/выключить интеграцию через UI.

Code smell: 0 в колонке OWNER — Owner проходит через bypass, но в декораторах не указан.

client · клиентский API17 файлов · 2 controller · 16 эндпоинтов

2 controller'а:

  • client.controller.ts (12) — авторизованные эндпоинты для клиента (профиль, история, бонусы)
  • client-public.controller.ts (4) — публичные (список доступных автоматов рядом, общая инфа)

После Фазы 3.6: этот модуль станет основой клиентского API после разделения users/clients таблиц.

import · импорт данных15 файлов · 23 эндпоинта

Назначение: массовый импорт сущностей из CSV/XLSX/PDF — машин, продуктов, контрагентов, начальных остатков.

Sprint G3 — HICON import с 3-уровневым дедупом и fuzzy matcher.

RBAC: только Owner/Admin/Manager — 23/23. Operator/Warehouse/Accountant — 0.

fcm · Firebase для mobile5 файлов · 7 эндпоинтов

Назначение: push-уведомления в mobile staff app через FCM.

Device tokens: хранятся в таблице device_tokens (миграция 1776300000000), привязаны к userId.

RBAC: симметрично — все 7 ролей могут регистрировать своё устройство (по 4-7 эндпоинтов).

web-push · браузерные push4 файла · 6 эндпоинтов

Browser push subscriptions через Service Worker. Использован для уведомлений Manager в админке (новые задачи, SLA breach, низкие остатки).

Recent change: push на manager + assigned tech при SLA breach в maintenance.

email и sms · listeners без HTTP4+4 файла · 0 эндпоинтов

Internal services. Подписаны на события email.send и sms.send, отправляют через провайдеров.

SMS: Eskiz.uz интеграция (стандарт для Узбекистана).

Email: SendGrid / Mailgun / SMTP — конфигурируется через env.

webhooks · subscriptions для внешних систем9 файлов · 9 эндпоинтов

Назначение: внешние системы (партнёры, BI-системы) могут подписаться на события VendHub. При срабатывании — POST на их URL с payload и подписью.

Reliability: BullMQ retry с exponential backoff если внешний URL не отвечает.

RBAC: только Owner/Admin — управляют webhook'ами. Manager не имеет доступа.

agent-bridge · API для autonomous agents7 файлов · 12 эндпоинтов

Назначение: специальный API для автоматизированных агентов (Claude, скрипты). Позволяет им работать с системой без полного UI.

AGENT_MODE: bypass JWT-проверки если в env AGENT_MODE=true и NODE_ENV !== production. Решение R2 в auth-аудите — нужно жёстко контролировать env.

Use case: Claude помогает оператору через CLI, AI-импорт данных.

data-parser · парсинг разных форматов10 файлов · 7 эндпоинтов

Назначение: универсальный парсер для CSV, XLSX, PDF выписок банка, реестров продаж. Используется import и payment-reports.

vhm24-integration · легаси-мост10 файлов · 6 эндпоинтов

Назначение: интеграция с легаси VHM24-repo (предыдущая система до VendHub OS). Используется во время переходного периода.

Не для пилота. Можно отключить или оставить как историческую поддержку.

✓Готовность группы

🟢 Готова при правильной настройке env. Каждая интеграция критически зависит от секретов в Railway. Без полного списка из LAUNCH_CHECKLIST половина работать не будет.

Стратегия пилота: начать с Telegram + email + web-push. SMS и FCM добавить через 1-2 недели если будут нужны.
