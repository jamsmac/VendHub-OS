▣Детали по модулям

orders · клиентские заказы7 файлов · 15 эндпоинтов

Назначение: жизненный цикл заказа от создания до завершения. Используется в Telegram Mini App.

Order entity

  • id, clientId (после Фазы 3.6 будет в clients table), machineId
  • status — DRAFT / PENDING_PAYMENT / PAID / DISPENSED / FAILED / REFUNDED
  • items[] — массив OrderItem с productId, quantity, price
  • amount, currency, discountAmount
  • paymentId — связь с Payment

Lifecycle events

order.created → order.paid → order.dispensed или order.failed → order.completed. На order.completed подписаны loyalty (баллы) и achievements (ачивки).

RBAC: Operator 9/15 — для machines операторам. Manager 13/15. Viewer 3/15 — основные read.

transactions · единый журнал транзакций14 файлов · 17 эндпоинтов

Назначение: единая запись каждой финансовой операции, агрегация платежей разных провайдеров.

Связь с payments: Payment — это «попытка оплаты», Transaction — «успешный финансовый факт». Не каждый Payment становится Transaction (если оплата провалилась, Transaction не создаётся).

RBAC: Owner/Admin/Manager — read+write, Accountant — read-only (10/17), Operator — 4/17 для своих машин.

sales-import · импорт из легаси-систем16 файлов · 9 эндпоинтов

Назначение: миграция данных из старых систем — HICON, BC, других CSV-форматов.

3-уровневый дедуп

Sprint G3 ввёл fuzzy-matcher для парсинга разных форматов с автоматическим mapping в внутренние сущности.

Use case: у клиента-франчайзи была старая система — импортируем 6 месяцев истории, дальше работаем с нуля.

complaints · жалобы и возвраты15 файлов · 28 эндпоинтов

Самый нагруженный эндпоинтами модуль группы. Покрывает workflow:

  1. Клиент создаёт жалобу через Mini App или кнопку «Возврат»
  2. Триаж — кто будет разбираться (Manager / Admin)
  3. Расследование — что случилось с автоматом, был ли товар выдан
  4. Решение — refund / replacement / decline
  5. Если refund — отдельный сервис complaints-refund.service.ts делает возврат через Payme/Click/Uzum
  6. Аудит-лог + уведомление клиента

Code smell: Operator 9/28 — может разбираться с жалобами на свои машины. Это правильный пользовательский опыт.

purchase-history · история клиента7 файлов · 10 эндпоинтов

Назначение: read-API для клиентского кабинета. Все покупки одного клиента с группировкой, фильтрацией, экспортом.

Используется в: Telegram Mini App «Мои покупки», web-кабинет клиента (после Фазы 3.6 будет в отдельном app).

purchases · закупки у поставщиков8 файлов · 6 эндпоинтов

Не путать с purchase-history. Purchases — это закупки сырья (зёрна, молоко, стаканчики) у contractors.

Sprint G2: создан с extension price_history — храним историю изменения закупочных цен.

Связь: contractors (поставщики), warehouse (приёмка на склад), inventory (увеличение остатка).
