**Date:** 2026-04-29

---

## TL;DR

- **84 модуля** API содержат защищённые `@Roles()` эндпоинты.
- **1067 декораторов `@Roles()`** распределены по этим модулям.
- **9 модулей** не содержат `@Roles()` совсем — это либо internal services без контроллера, либо публичные эндпоинты (health, metrics, monitoring, public-tenant). Все объяснимы.
- **Найдены 3 кодовых неконсистентности** (см. раздел «Code smells»).

## Критическое уточнение про Owner

В матрице ниже колонка `OWN` (Owner) показывает **только явные упоминания** роли в декораторах. Это **не** значит, что в модулях с `0` в колонке `OWN` владелец не имеет доступа.

В коде [`apps/api/src/modules/auth/guards/roles.guard.ts`](../../apps/api/src/modules/auth/guards/roles.guard.ts) есть глобальный bypass:

```ts
// Owner has access to everything
if (user.role === UserRole.OWNER) {
  return true;
}
```

Это значит: **Owner проходит через любой `@Roles(...)` фильтр без явного указания**. То есть строки матрицы со значением `0` в колонке `OWN` — это просто стиль написания кода, не реальная блокировка владельца.

**Следствие:** Owner всегда имеет доступ ко всему, что доступно через REST API. Это архитектурное решение — упрощает написание декораторов, но усложняет аудит «кто куда вошёл из реальных Owner-аккаунтов». Все действия Owner логгируются в `audit` модуле.

## Как читать матрицу

- Колонка `EP` — сколько эндпоинтов в этом модуле защищены `@Roles`.
- Колонки `Own / Adm / Mgr / Acc / Wrh / Opr / Vwr` — на скольких из этих эндпоинтов роль явно разрешена.
- Стиль `enum` — модуль использует `UserRole.ADMIN`, `string` — `"admin"`, `MIXED` — оба стиля внутри одного модуля (плохо, см. ниже).

## Code smells (3 находки)

### 1. Mixed style — модули с обоими видами @Roles

`loyalty`, `referrals`, `reports` используют **и** `UserRole.ADMIN`, **и** `"admin"` в одном модуле. Это значит, что разные разработчики писали разные части и не унифицировали стиль.

**Риск:** низкий технический, но опечатка типа `"administer"` (вместо `"admin"`) пройдёт TypeScript-проверку и фактически закроет эндпоинт от всех ролей. С enum такой опечатки быть не может.

**Рекомендация:** в рамках post-launch refactoring перевести всё на enum.

### 2. Нулевая колонка OWN в 3 модулях

`fiscal`, `analytics`, `integrations` имеют 0 в колонке Own. Это не блокировка (Owner проходит через global bypass), но **читать матрицу/audit-логи становится сложнее**.

**Риск:** средний для compliance — трудно доказать аудитору, что Owner действительно может работать с фискалом, не глядя в код guard'а.

**Рекомендация:** в рамках refactoring добавить `"owner"` явно во все @Roles. Это тривиальная правка.

### 3. Подозрительная нулевая колонка ACCOUNTANT в `inventory`, `equipment`, `integrations`

В `inventory` (14 эндпоинтов) бухгалтер имеет 0 доступа. Это может быть правильно (склад — не его дело), но может быть и ошибкой — в проде бухгалтер не сможет посмотреть остатки, чтобы провести списание.

**Риск:** средний продуктовый. Зависит от того, как у вас реально работает бухгалтер.

**Рекомендация:** обсудить с реальным бухгалтером, нужен ли ему read-доступ к складу.

## Что это даёт тебе как владельцу

1. **Карта прав в одном месте.** Когда придёт новый сотрудник, ты сразу видишь, какую роль ему дать, чтобы он мог делать свою работу и не больше.
2. **Compliance.** Если придёт аудитор и спросит «кто может посмотреть платежи?» — у тебя есть готовый ответ.
3. **Базис для будущих решений.** Когда добавляется новый модуль, можно проверить «не выпала ли роль» — например, ты по аналогии увидишь, что Operator должен иметь доступ к новому подобию tasks.
