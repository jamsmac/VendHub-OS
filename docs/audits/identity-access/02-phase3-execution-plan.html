Команда — какие агенты и скиллы привлекаем

У проекта уже 6 кастомных агентов и 15 VHM24-скиллов. Распределение этих ресурсов по конкретным задачам Фазы 3.

агент — самостоятельный исполнитель с собственным контекстом
скилл — паттерны и инструкции, применяются Claude'ом

Готовые агенты в проекте

АгентМодельЗона ответственности
module-migrator opus Перенос NestJS-модулей. Изначально для VHM24→VendHub, но идеален для любой переделки модулей: сейчас — для миграции users на новую схему и разделения users/clients.
db-migration-helper sonnet TypeORM-миграции: создание новых таблиц user_permissions, clients, client_organizations. Проверка консистентности entity ↔ схемы БД.
build-verifier sonnet После каждого PR — проверка что весь монорепо компилируется. Без этого нельзя мерджить.
production-quality-guardian opus Финальная проверка перед запуском Фазы 3 в прод. Ищет скрытые баги, ошибки конфигурации, regression'ы по функциональности.
health-check sonnet SRE-диагностика. Запускается во время и после миграции — проверить что Docker-сервисы, БД, API живы.
dependency-audit sonnet Аудит безопасности. Запускается раз в 2 недели — проверить новые CVE, устаревшие пакеты.

База знаний — что писать и куда сохранять

Чтобы Claude (любая сессия) и команда могли подхватить работу без перепрочитывания всего кода. У проекта уже есть фундамент: CLAUDE.md на 1111 строк и папка docs/audits/. Расширяем её.

Структура knowledge base

1. Корневой контекст · быстрый въезд в проект ›
  • CLAUDE.md — общий контекст проекта. Уже есть, 1111 строк. Не трогаем кроме крайней нужды.
  • docs/HANDOFF.md — последний снимок состояния. Обновляется в конце каждой сессии или спринта.
  • docs/audits/MODULE_AUDIT_INDEX.md — журнал аудита модулей и принятых решений. Каждая сессия читает его первым.
  • RESOURCES.md — список людей, аккаунтов, доступов. Уже есть.
2. Архитектурные решения · ADR-формат ›

Для каждого важного выбора записываем ADR (Architecture Decision Record) в docs/adr/:

  • ADR-001 — переход на 4 staff + Client + конструктор
  • ADR-002 — разделение users и clients (две таблицы, два audience)
  • ADR-003 — Owner — singleton, проверка через user_id
  • ADR-004 — мета-галочки can_create_* как permissions, не как столбцы users
  • ADR-005 — стратегия миграции декораторов @Roles → @Permissions

Формат ADR: контекст / альтернативы / решение / последствия. Один ADR — один файл, не редактируется задним числом, только дополняется новым ADR при пересмотре.

3. UX-спецификации · перед каждым экраном ›

В docs/specs/:

  • spec-users-list.md — экран списка пользователей

  • spec-create-user-wizard.md — wizard создания
  • spec-machine-attachment.md — экран закреплений
  • spec-client-cabinet.md — кабинет клиента

Формат: цель / personas / user stories / компоненты / состояния / edge cases / acceptance criteria / макет (можно ссылка на Figma или ASCII-mockup).

Использование: vhm24-ux-spec создаёт черновик, человек ревьюит, скилл vhm24-ui-generator читает и генерирует код.

Каждый отчёт: назначение / связи / схема / API / RBAC / тесты / риски / готовность. Этот формат уже стандартизован.

Workflow — как организовать работу день за днём

Чтобы Фаза 3 не превратилась в хаос. 8 подэтапов структурированы в 5 спринтов, каждый спринт занимает примерно 1 неделю. После каждого — деплой на staging, ревью, потом в прод.

Главный принцип: миграция Auth/RBAC — это не место для героических многоликих PR'ов. Каждый шаг — маленький, изолированный, обратимый. Сначала ставим параллельно с существующим, проверяем, потом удаляем старое.

5 спринтов Фазы 3

Спринт A — фундамент в коде · 3.1 + 3.2 ›

Цель: PermissionsGuard и assertCanGrant работают параллельно с @Roles. Никаких изменений снаружи.

Команда: vhm24-auth-rbac + vhm24-api-generator + vhm24-testing + build-verifier

Шаги:

  1. ADR-005 (стратегия параллельных декораторов) — engineering:architecture
  2. Расширить RbacService методом assertCanGrant(grantorId, permissionKey)
  3. Создать PermissionsGuard в common/guards
  4. Добавить unit-тесты на оба механизма (~30 кейсов)
  5. На один тестовый эндпоинт повесить @Permissions параллельно с @Roles — проверить что оба гарда срабатывают
  6. build-verifier + e2e smoke

Acceptance: RBAC canary тест проходит, новый PermissionsGuard покрыт тестами, существующая функциональность не сломана.

Спринт B — мета-галочки и сокращение enum · 3.3 + 3.4 ›

Цель: can_create_managers / can_create_employees как permissions. UserRole enum сокращён с 7 до 4. Скрипт миграции существующих пользователей готов и протестирован на staging.

Команда: module-migrator + db-migration-helper + vhm24-db-expert + vhm24-testing

Шаги:

  1. ADR-004 (мета-галочки как permissions, не как поля)
  2. Добавить два permission в seed: users:create_manager, users:create_employee
  3. Скрипт миграции: для каждого пользователя проставить permissions на основе текущей роли (по матрице из 01-role-matrix.md)
  4. Запустить на staging-копии прода. Сравнить «до» и «после» доступы для 100 случайных пользователей.
  5. Сократить enum UserRole — удалить OPERATOR, WAREHOUSE, ACCOUNTANT, VIEWER
  6. Обновить ROLE_HIERARCHY и ROLE_LABELS
  7. production-quality-guardian финальный аудит

Acceptance: на staging все эндпоинты доступны тем же пользователям, что и до миграции. RBAC canary 0 нарушений.

Спринт C — Admin привязка к организации · 3.5 ›

Цель: Admin теперь жёстко привязан к одной организации. Может видеть только свои данные. Только Owner видит всё.

Команда: module-migrator + vhm24-auth-rbac

Шаги:

  1. Перебрать запросы, которые возвращают «все организации» — заменить на фильтр по user.organizationId, кроме Owner
  2. OrganizationGuard — убедиться что injected org теперь обязательный

Daily / weekly ритм

⏱
Каждый день начинается с Чтения MODULE_AUDIT_INDEX.md + последнего HANDOFF. Это 5 минут — но даёт полный контекст где остановились.
⏱
Каждая задача начинается с UX-спека или ADR (если затрагивает архитектуру). Только потом — код.
⏱
Каждый PR заканчивается build-verifier + vhm24-qa-review + engineering:code-review. Тройная проверка — миграция auth не место для срезания углов.
⏱
Каждый спринт заканчивается Деплой на staging → smoke-тесты → ревью → деплой в прод. Между staging и прод — минимум 24 часа на наблюдение.
⏱
Каждые 2 недели dependency-audit + проверка что новые CVE не появились в зависимостях.
⏱
Перед самым последним PR Фазы 3 production-quality-guardian делает финальный аудит. Без его OK не мерджим.

Готовность работать в этом ритме

У тебя как Owner'а проекта в этом workflow роль не «писать код», а «делать решения»:

  • Утверждать ADR — это раз в неделю короткий документ на 1 страницу
  • Утверждать UX-спеки перед генерацией экранов — раз в неделю 1-2 спека
  • Принимать staging-деплой — пройтись по 5-10 сценариям руками, дать «зелёный»
  • Финальное согласие на прод-деплой

Программирование делают агенты + команда + Claude. Архитектурные решения и продуктовые приоритеты — твоя зона.
