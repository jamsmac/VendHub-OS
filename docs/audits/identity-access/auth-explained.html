1. Что такое auth одной фразой

Это охранник на входе в систему. Решает кого впускать, под каким бейджиком, и закрывает 129 защищённых дверей в остальных модулях.

Когда оператор открывает страницу /dashboard, мобильное приложение или Telegram Mini App — первый кто проверяет его бейджик, это auth. Если бейджик битый, просрочен, отозван или подделан, доступа не будет ни к одному защищённому ресурсу.

Auth не управляет самими сотрудниками (это users) и не выдаёт приглашения (это invites). Он умеет только три вещи: проверить кто ты, выдать тебе бейджик и отозвать бейджик когда нужно.

3. Семь ролей и иерархия

Каждый сотрудник имеет одну роль. У ролей есть «вес» — чем выше, тем больше прав. Owner получает доступ ко всему автоматически, не нужно перечислять его в каждой проверке.

Owner — Владелец100 / 100 — полный доступ ко всем организациям
Admin — Администратор90 / 100 — администратор одной организации
Manager — Менеджер70 / 100 — управление командой и задачами
Accountant — Бухгалтер50 / 100 — платежи, отчёты, фискализация
Warehouse — Кладовщик40 / 100 — склад, остатки, приёмка
Operator — Оператор30 / 100 — маршруты, визиты к автоматам
Viewer — Наблюдатель10 / 100 — только просмотр

Как это работает в коде

На любую защищённую функцию ставится метка-список разрешённых ролей. Например:

@Roles('admin', 'accountant')   // только админ и бухгалтер
@Roles('owner')                  // только владелец
@Roles(...VIEWER_AND_ABOVE)      // все 7 ролей

Owner проходит через любой такой фильтр без явного перечисления — это «костыль» по дизайну, чтобы не указывать его в каждой проверке.

Имена ролей — на русском и узбекском — хранятся отдельно (ROLE_LABELS) и используются в интерфейсе, чтобы не показывать пользователю английские слова вроде warehouse.

4. Четыре способа войти

Все четыре способа в итоге выдают одинаковый бейджик-токен. Отличается только то, как сотрудник доказывает «я — это я» в самом начале.

A. Email + пароль

Классика. Лимит — 10 попыток с одного IP в минуту. После 5 подряд неверных паролей аккаунт блокируется на 30 минут.

Пароль хранится не как текст, а как «отпечаток» (bcrypt с фактором 12) — даже если базу украдут, пароли не восстановить.

B. Через Telegram Mini App

Когда клиент открывает Mini App в Telegram, Telegram сам присылает данные пользователя с цифровой подписью.

Сервер берёт эти данные, пересчитывает подпись своим ключом бота (TELEGRAM_BOT_TOKEN) и сравнивает. Совпало — пускаем; не совпало — отказ.

C. По коду-приглашению

Менеджер заходит в систему, нажимает «Пригласить оператора», задаёт роль и организацию, получает уникальную ссылку с кодом.

Сотрудник открывает ссылку — попадает на форму регистрации, где нужно только задать пароль (или связать Telegram-аккаунт). Роль и организация уже зашиты в код.

D. Двухфакторка (2FA)

Опциональный второй слой поверх любого из A, B, C. Можно включить отдельно для каждого пользователя.

После ввода пароля система просит шестизначный код из приложения (Google Authenticator, 1Password, Aegis). Плюс выдаются 10 одноразовых резервных кодов на случай потери телефона.

6. Что происходит при входе пошагово

Пример: оператор Андрей открывает админку и вводит email и пароль. Что делает сервер за следующую секунду.

Браузер Андрея → API
1. Отправляет POST /auth/login с email и паролем

Запрос идёт по HTTPS, пароль передаётся в теле запроса.

API: Throttler
2. Проверка лимита частоты

Если с этого IP за последнюю минуту уже было 10 попыток входа — отказ 429.

AuthService → users
3. Поиск пользователя по email

Если нет — записывается «провальная попытка», возвращается 401. Сообщение специально неинформативное, чтобы не подсказывать атакующему «email есть, но пароль не тот».

AuthService
4. Проверка блокировки

Если у пользователя 5+ провалов за последние 30 минут — отказ «аккаунт заблокирован».

bcrypt
5. Сверка пароля

Введённый пароль прогоняется через bcrypt с фактором 12, сравнивается с хешем из базы.

AuthService
6. Проверка 2FA

Если у пользователя включён второй фактор, login не возвращает токен — возвращает короткоживущий challengeToken. Клиент покажет форму «введите код из Authenticator», и потом отдельным запросом вызовет /2fa/complete.

AuthService → база
7. Создание сессии

В таблице user_sessions создаётся запись: id сессии, user_id, IP, user-agent, expiresAt = +7 дней. Сессии записываются для того, чтобы можно было «разлогинить» с конкретного устройства, не трогая остальные.

JwtService
8. Подпись двух токенов

Подписывается access-токен (живёт 15 минут, для каждого запроса) и refresh-токен (живёт 7 дней, нужен только чтобы обновлять access). Подпись делается секретом из переменной JWT_SECRET алгоритмом HS256.

CookieService
9. Установка куков в ответе

Оба токена кладутся в куки vendhub_access_token и vendhub_refresh_token. Куки помечены HttpOnly (JavaScript на странице их не видит — защита от XSS), Secure (только HTTPS) и SameSite (защита от CSRF).

MetricsService
10. Счётчик +1

В метриках инкрементируется «успешный логин» — для статистики и мониторинга.

Браузер Андрея
11. Получает ответ + сохраняет куки

Андрей видит дашборд. На каждом следующем запросе браузер автоматически прикладывает куки.

7. Жизнь бейджика-токена

У каждого вошедшего пользователя есть два токена с разной ролью.

Access-токен — «дневной пропуск»

  • Живёт 15 минут
  • Кладётся в куку vendhub_access_token
  • Прикладывается к каждому запросу к API
  • Если он битый или просрочен — 401
  • Внутри: id пользователя, email, роль, organizationId, sessionId, jti (уникальный id этого токена)

Refresh-токен — «билет на новый пропуск»

  • Живёт 7 дней
  • Используется только для одного — получить новый access
  • Когда access истекает, фронтенд тихо вызывает /auth/refresh и получает свежую пару
  • При каждом refresh оба токена меняются (token rotation)
  • Старый refresh после использования становится невалидным

Как отзывается токен

Токены подписаны и не хранятся на сервере. Чтобы один конкретный токен «уволить» досрочно, есть два механизма:

  • Чёрный список по jti — id конкретного токена кладётся в Redis на 15 минут (срок access). При следующей проверке guard видит «этот id в blacklist» и отказывает.
  • Чёрный список по пользователю — после смены пароля все токены, выпущенные раньше определённого момента, считаются недействительными. Это закрывает кейс «у меня украли ноут — поменяю пароль и пусть все токены сгорят».

8. Что хранится в базе

Auth работает с пятью таблицами. Сами таблицы определены в модуле users, но все они обслуживают auth-сценарии.

users — сами сотрудники и клиенты

  • id — UUID
  • email — для логина
  • password — bcrypt-хеш
  • role — одна из 7
  • organizationId — какая организация
  • status — active / inactive / blocked
  • telegramId — для входа через Telegram
  • telegramUsername
  • firstName, lastName
  • loginAttempts — счётчик провалов
  • passwordChangedAt — для blacklist по iat
  • createdAt, updatedAt, deletedAt

user_sessions — где сейчас залогинен

  • id — UUID сессии
  • userId — кто
  • jti — id текущего refresh-токена
  • ipAddress — откуда
  • userAgent — какой браузер/устройство
  • expiresAt — когда истекает
  • isRevoked — отозвана ли вручную

two_factor_auth — настройки 2FA

  • userId — для кого
  • secret — секрет для TOTP-кода
  • enabled — включена ли
  • backupCodes — массив одноразовых кодов

password_reset_tokens — ссылки «забыл пароль»

  • userId
  • tokenHash — хеш токена из ссылки
  • expiresAt — обычно час-два
  • isUsed — использован ли

login_attempts — журнал попыток входа

  • email
  • ipAddress
  • success — true/false
  • createdAt

Auth не создаёт эти таблицы напрямую — они появились в самой первой миграции 1700000000000-Init вместе со всем фундаментом.

9. Три ночные уборки

Auth-сервис подписан на расписание — три задачи запускаются автоматически по ночам по Ташкенту.

03:00 каждый день

Чистка просроченных ссылок для сброса пароля. Все записи в password_reset_tokens, у которых expiresAt в прошлом — soft-delete.

04:00 каждый день

Чистка просроченных сессий. Удаляются записи из user_sessions, где истёк срок или сессия была вручную отозвана.

05:00 каждый день

Чистка журнала входов старше 30 дней. Удаляется жёстко (не soft-delete) — этого требует GDPR, такие записи содержат email + IP и не должны храниться вечно.

Внимание. В Docker-образе API сейчас нет пакета tzdata, поэтому часовой пояс Asia/Tashkent внутри контейнера не распознаётся — задачи фактически запустятся в 03/04/05 по UTC, что соответствует 08/09/10 утра по Ташкенту. Это пункт 5 в списке рисков ниже.

11. Защита от атак

Что у системы есть «из коробки» против типичных атак на логин.

Brute force паролей

Защита: 5 неверных попыток подряд → блокировка аккаунта на 30 минут. Плюс лимит 10 попыток с одного IP в минуту через ThrottlerModule.

Кража токена (XSS на странице)

Защита: токены лежат в HttpOnly-куках, JavaScript на странице их не видит. Плюс короткий срок жизни access (15 мин).

CSRF (атака с чужого сайта)

Защита: у куков выставлен SameSite. По умолчанию lax — подходит для same-origin, но для cross-origin требуется none (см. риск 3).

Кража базы данных

Защита: пароли в bcrypt с фактором 12 — 2^12 = 4096 итераций хеша. Восстановить пароль из хеша возможно только через очень дорогой перебор.

Угнанный access-токен

Защита: при логауте jti токена кладётся в blacklist. При смене пароля все токены пользователя становятся невалидными по iat.

IDOR в 2FA

Защита: поток 2FA не передаёт userId напрямую. После логина с правильным паролем выдаётся короткоживущий подписанный challengeToken, и только он принимается на втором шаге.

Подделка Telegram-входа

Защита: данные от Telegram проверяются HMAC-подписью с использованием секретного TELEGRAM_BOT_TOKEN. Без правильного токена сервер не может ни принять запрос, ни подделать его.

Replay использованной reset-ссылки

Защита: у токена сброса пароля есть isUsed. После использования — отказ. Плюс срок жизни 1-2 часа.

12. Связи с другими модулями

Auth — самый связанный модуль системы. Сам он опирается на 4 модуля, и от него зависят почти все остальные.

На что опирается auth (без них он не работает)

users
«Отдел кадров». Auth идёт сюда проверить «такой email есть? пароль совпадает? пользователь активен?». Сам auth не хранит сотрудников.
invites
«Бюро пропусков». При регистрации по коду auth спрашивает «такой код есть? не использован? в какую организацию и роль ведёт?».
metrics
«Счётчики». После каждого логина auth говорит «прибавь к счётчику успешных или провальных логинов». Используется для алертов.
email
«Почтальон». Через событие email.send auth просит выслать письмо со ссылкой для сброса пароля. Связь слабая — через события, а не через прямой импорт.

Кто использует auth (129 файлов в системе)

Каждый защищённый эндпоинт в любом модуле помечен декораторами @UseGuards(JwtAuthGuard) и @Roles(...) — это артефакты auth-модуля. Когда приходит запрос, NestJS сначала вызывает JwtAuthGuard (проверить бейджик), затем RolesGuard (проверить роль).

14. Пять рисков перед запуском

Это не дефекты кода, а настройки сервера, которые надо прописать в Railway. Все вместе — около 30 минут работы.

R1. Печать на бейджах — JWT_SECRET

В коде есть фолбэк: если переменной нет — токены подписываются пустой строкой. JwtStrategy явно падает при пустом секрете, но JwtService молча подписывает.

Действие: прописать JWT_SECRET в Railway env (≥32 случайных байта)

R2. Режим «тестов» — AGENT_MODE

В JwtAuthGuard есть тестовый режим, который пропускает запросы без проверки. Он отключается, только если NODE_ENV=production.

Действие: прописать NODE_ENV=production в Railway env

R3. SameSite куки для cross-origin

Web-админка и API будут на разных Railway-доменах. По умолчанию SameSite=lax, что не передаёт куки на cross-origin POST. Логин на проде сломается.

Действие: прописать COOKIE_SAME_SITE=none в Railway env API

R4. TELEGRAM_BOT_TOKEN

Без него ни один Telegram-вход не пройдёт — HMAC-подпись не сверится.

Действие: прописать TELEGRAM_BOT_TOKEN из @BotFather

R5. Часовой пояс в контейнере — tzdata

Образ node:20-alpine не содержит tzdata. @Cron('0 3 * * *', { timeZone: 'Asia/Tashkent' }) фактически работает в UTC. Три ночные задачи запустятся в 08, 09, 10 утра по Ташкенту, посреди рабочего дня.

Действие: добавить «RUN apk add --no-cache tzdata» в production stage Dockerfile

15. Чеклист настроек перед запуском на Railway

Эти переменные надо завести в Railway → Service «api» → Variables.

JWT_SECRET Длинная случайная строка. Сгенерировать: openssl rand -base64 48. Использовать одну и ту же на всех инстансах api, не менять без причины.
NODE_ENV Значение: production. Без этого может остаться включённым AGENT_MODE и cookie.secure будет false.
COOKIE_SAME_SITE Значение: none. Чтобы куки между разными Railway-доменами web и api передавались.
TELEGRAM_BOT_TOKEN Получить у @BotFather в Telegram. Не логировать, не коммитить.
JWT_EXPIRES_IN Значение по умолчанию 15m подходит для прода. Можно не задавать.
SESSION_DURATION_DAYS Значение по умолчанию 7 подходит. Меньше — пользователи будут чаще логиниться, больше — выше риск угнанной сессии.
PASSWORD_MIN_LENGTH (опционально) По умолчанию 8. Для серьёзных контуров можно поставить 12.
tzdata в Dockerfile Это правка кода, а не env. Добавить «RUN apk add --no-cache tzdata» в production stage apps/api/Dockerfile.
Smoke-тест после деплоя 1) POST /auth/login с тестовыми кредами, проверить что получили cookie. 2) GET /auth/me с этой cookie — 200 + профиль. 3) POST /auth/logout — 204. Если шаги 1-2-3 проходят — auth работает.
