# Принятое решение: упрощение модели ролей до 4 staff + 1 client + конструктор прав

**Дата:** 2026-04-29
**Принято:** Jamshid (Owner)
**Статус:** Зафиксировано как продуктовое направление. Реализация — после стабилизации пилота.

---

## Решение в одной фразе

Сократить 7 жёстких ролей до **4 staff-ролей** (Owner, Admin, Manager, Employee) + **отдельная роль Client** для покупателей с собственным кабинетом. Добавить **per-user конструктор прав** для staff — каждый уровень может выдавать своим подчинённым только те права, которые есть у него самого. «Наблюдатель» — это пресет шаблона внутри Employee, не отдельная роль.

## Два изолированных мира

Staff и Client живут в разных пространствах, не пересекаются:

- **Staff** — внутренняя команда. Owner / Admin / Manager / Employee. Логин через email+пароль (или Telegram-привязка для удобства). Привязка к организации.
- **Client** — покупатели. Логинятся сами через Telegram Mini App или phone+OTP. Свой кабинет: профиль, покупки, бонусы, ачивки, рефералы. Не видят и не могут попасть в staff-интерфейс. Могут быть привязаны к нескольким организациям одновременно (если покупают у разных франчайзи).

## Жёсткие правила

1. **Owner — один на всю систему. Это Jamshid.** Никаких других owner'ов, никогда. В коде должна быть либо проверка по конкретному user_id, либо явное правило «owner создаётся только при первом запуске и единственный».
2. **Admin привязан к одной организации.** Не «все организации» как сейчас. Для франчайзи-структуры один админ = одна организация.
3. **Каждый создаваемый пользователь получает права галочками.** При создании по умолчанию максимум того, что есть у создателя; создатель может снять часть галочек.
4. **Нельзя выдать право, которого у тебя самого нет.** Это enforcement на уровне API: попытка установить permission, которого нет у granter'а, отбивается.
5. **Admin может создавать и Manager, и Employee напрямую.** Не обязательно строгая цепочка. Admin может нанять оператора без посредника-менеджера.
6. **«Может создавать подчинённых» — отдельная мета-галочка.** Это не функциональное право на модуль, а способность создавать пользователей нижестоящих уровней. Manager без этой галочки — исполнитель, с галочкой — может нанимать сотрудников. Admin по умолчанию имеет обе галочки (создавать менеджеров и создавать сотрудников), их можно отключить отдельно.
7. **Делегирование одно и то же на каждом уровне.** Правило «только из своих» применяется одинаково: Admin раздаёт из всех прав орг, Manager раздаёт из подмножества, которое Admin ему дал.

## Структура

| Роль         | Может создавать                                                           | Видит                                                          | Получает по дефолту                                                                                 |
| ------------ | ------------------------------------------------------------------------- | -------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Owner**    | Admins (только)                                                           | Все организации                                                | Всё                                                                                                 |
| **Admin**    | Managers + Employees (две отдельные галочки)                              | Только свою организацию                                        | Все права своей орг (кроме owner-функций) + обе галочки создания                                    |
| **Manager**  | Employees — если Admin включил мета-галочку «может создавать сотрудников» | Своих сотрудников + общие данные орг по permissions            | Подмножество прав admin'а, выставленное при создании. Мета-галочка создания по умолчанию выключена. |
| **Employee** | Никого                                                                    | Свои задачи и работу + общие данные орг по permissions         | Подмножество прав создателя (Admin или Manager)                                                     |
| **Client**   | — (саморегистрация)                                                       | Только свои данные: профиль, покупки, бонусы, ачивки, рефералы | Базовый набор клиент-permissions, индивидуально настраивается под программу лояльности орг          |

### Шаблон «Наблюдатель» внутри Employee

Это не отдельная роль, а **именованный пресет** в UI создания сотрудника. При выборе шаблона «Наблюдатель» автоматически проставляются галочки:

- Все 8 категорий — только `read`, никаких `write` / `delete`.
- Мета-галочка `can_create_employees` — выключена.

Кому это подходит: инвестору, аудитору, новичку на испытательном сроке, внешнему бухгалтеру-консультанту.

В UI это выглядит как кнопка-пресет рядом с другими шаблонами («Оператор», «Кладовщик», «Бухгалтер»). Создатель может потом снять/добавить отдельные галочки поверх шаблона.

### Мета-разрешения на создание

Это особый класс permissions, отделённый от функциональных прав:

- `can_create_managers` — есть только у Owner (для Admin) и у Admin (опционально, при создании Admin). По дефолту у Admin включено.
- `can_create_employees` — есть у Admin (по дефолту) и у Manager (только если Admin явно включил при создании).

Manager без `can_create_employees` — это «специалист»: работает с данными в рамках своих прав, но команду не нанимает.
Manager с `can_create_employees` — это «руководитель команды».

## Категории прав (шаблоны для UI)

Это группы галочек, чтобы не показывать 200 чекбоксов:

1. **Поле** — routes, tasks, maintenance, work-logs, incidents
2. **Склад** — inventory, warehouse, storage, containers, stock-movements, batch-movements, opening-balances
3. **Деньги** — payments, telegram-payments, fiscal, billing, payment-reports, cash-finance, payout-requests
4. **Каталог** — products, categories, directories, references, custom-fields
5. **Лояльность** — loyalty, achievements, quests, complaints, promo-codes, referrals
6. **Отчёты** — reports, analytics, ai, audit (read), investor
7. **Машины** — machines, equipment, alerts, predictive-refill, geo, locations
8. **Админка** — users, rbac, security, settings, organizations (только Admin)

Внутри каждой категории — read / write / delete как отдельные галочки или агрегированный «всё включено».

## Технический долг: разделение users и clients

В текущем коде клиенты живут в той же `users` таблице, что и staff — это техдолг, который мешает чистому разделению.

При миграции:

- Staff остаётся в `users`.
- Клиенты переносятся в отдельную таблицу `clients` со своей структурой полей (имя, телефон, telegram_id, email опционально, реферальный код, organizationIds[]).
- Связь client → organization через таблицу-промежуточник `client_organizations(client_id, organization_id, joined_at, status)` — клиент может покупать у нескольких франчайзи.
- Логин клиента — через отдельные эндпоинты `/auth/client/telegram` и `/auth/client/phone-otp`. Не пересекается со staff-логином.
- JWT-токен клиента имеет другой `audience` claim — `vendhub-clients`. Guards проверяют, что staff-эндпоинты принимают только `vendhub-users`, и наоборот.

## Технический план реализации

> **Примечание про термины.** В этом документе используется слово «Этап» для подзадач внутри миграции. В общем плане проекта (`01-role-matrix.html`) есть «Фаза 3 — миграция модели ролей» — это контейнер для всех 8 этапов ниже. Не путать.
>
> **Примечание про готовность.** При проверке кода обнаружено, что значительная часть фундамента уже есть: динамические роли, гранулярные permissions, machine-access с 5 типами доступа, audit log. Не строим с нуля — доинтегрируем существующее.

### Этап 1 — PermissionsGuard и интеграция с существующим RBAC

- Создать `PermissionsGuard` в `common/guards`, читающий permissions через существующий `RbacService`.
- Параллельно с `@Roles` ввести массовое использование `@Permissions`.
- Оба гарда работают одновременно — миграция постепенная, без big-bang.

### Этап 2 — Правило «нельзя выдать чего нет»

- В `RbacService` добавить `assertCanGrant(grantorId, permissionKey)`.
- Попытка дать роль или permission, которых нет у granter'а, отбивается на API уровне.
- Это техническое усиление, которого сейчас нет.

### Этап 3 — Мета-галочки can_create_managers / can_create_employees

- Добавить в seed таблицы `permissions` две системные записи: `users:create_manager`, `users:create_employee`.
- Включаются индивидуально при создании Admin / Manager.
- Owner всегда имеет обе.

### Этап 4 — Сокращение enum UserRole с 7 до 4

- Удаление OPERATOR, WAREHOUSE, ACCOUNTANT, VIEWER из основного enum.
- Они становятся именованными пресетами галочек, не ролями.
- Скрипт миграции пересчитывает каждого существующего пользователя в нужный шаблон Employee.
- Owner — добавить уникальный constraint «единственный».

### Этап 5 — Admin привязан к одной организации

- Сейчас Admin может видеть все организации. Переделать так, чтобы Admin был жёстко связан с одной через `user.organizationId`.
- Только Owner видит все организации.
- Тесты на cross-org изоляцию.

### Этап 6 — Разделение users и clients

- Новая таблица `clients` со своей структурой полей (имя, телефон, telegram_id, реферальный код).
- Связь client → organization через таблицу-промежуточник `client_organizations`.
- Логин клиента — отдельные эндпоинты `/auth/client/telegram` и `/auth/client/phone-otp`.
- JWT-токен клиента имеет другой `audience` — `vendhub-clients`. Гарды отбивают чужой токен.

### Этап 7 — UI единой панели управления людьми

- Список всех пользователей орг с фильтрами.
- Кнопка «Добавить» (Manager / Employee, видна по мета-галочкам).
- Карточка пользователя с галочками 8 категорий + мета-галочки + закреплённые машины.
- История изменений прав.
- Шаблоны «Оператор», «Кладовщик», «Бухгалтер», «Наблюдатель» как кнопки-пресеты.

### Этап 8 — Миграция декораторов с @Roles на @Permissions

- 1067 декораторов перевести поэтапно. По одному модулю за PR, оба декоратора параллельно работают.
- После полной миграции старые `@Roles` и неиспользуемые роли в enum удаляются.

### Группировка в спринты

В `02-phase3-execution-plan.html` 8 этапов сгруппированы в 5 спринтов: A=1+2, B=3+4, C=5, D=6, E=7+8. Это для удобства разработки, не отдельная семантика.

## Когда делать

**НЕ перед пилотом.** Перед запуском 23 машин на Railway — оставить текущие 7 ролей. Текущая модель работает, всё протестировано (1067 декораторов, 121 тест в auth).

**Когда начинать миграцию:** после стабилизации пилота, когда выполняется одно из условий:

- Появится второй админ на другую организацию (франчайзи).
- Размер команды превысит 10 человек.
- Появится чёткая потребность в индивидуальных правах для разных менеджеров.

Сроки разработки специально не указаны — реализация постепенная, по мере появления потребности и доступности команды. Каждый из 8 этапов изолирован и обратим, можно остановиться в любой момент без поломки системы.

## UI: единая панель управления людьми (out of scope, но зафиксировано)

Это не задача аудита, но решение принято: всё управление пользователями — в **одной панели админа**, а не разрозненно по экранам «Менеджеры», «Операторы», «Бухгалтеры».

Что должно быть в панели:

- Список всех пользователей организации (для Admin) или своей команды (для Manager).
- Фильтр по роли (Manager / Employee), статусу, тегу шаблона.
- Кнопка «Добавить пользователя» с выбором уровня (Manager / Employee). Если у создающего нет соответствующей мета-галочки — кнопка не показывается.
- Карточка каждого пользователя — раскрывающаяся, внутри: галочки прав по 8 категориям + мета-галочки создания + кнопки «Заблокировать», «Сменить пароль», «Перевести в другую категорию».
- Аудит-лог изменений прав внутри карточки: «Алишер: 28.04.2026 включил Деньги:read для Ивана».

Детальный UX — на Этапе 7 миграции, отдельная задача дизайна. Mockup'ы экранов сохранены в `02-phase3-execution-plan.html` вкладка «Что станет».

## Открытые вопросы для будущего обсуждения

1. Что делать с заблокированным пользователем — снимать ли права полностью, или оставлять как archive?
2. Как обрабатывать случай «admin удалил у себя право, которое раздавал менеджеру» — каскадный отзыв или сохранение?
3. Должны ли права наследоваться (если admin даёт право manager'у, то его сотрудники получают тот же по умолчанию)?
4. Нужен ли «approval flow» для чувствительных действий (см. идеальную модель в `01-role-matrix.html`, фаза 3)?
5. Если manager без `can_create_employees` создан, а потом admin включил эту галочку — нужно ли явное уведомление manager'у, или он просто увидит новую кнопку при следующем входе?
6. Может ли Owner забрать у Admin мета-галочку «может создавать менеджеров» и оставить только «может создавать сотрудников»? Если да — что происходит с уже созданными им менеджерами?

Эти вопросы решаются на детализации Этапа 1.
