Найденные проблемы

Code smell #1: смешанные стили @Roles (3 модуля) — низкий риск ›

Кто: loyalty, referrals, reports.

В одном модуле часть декораторов использует UserRole.ADMIN (енам, защищённый TypeScript), часть — "admin" (строку, без проверки).

Чем рискует: опечатка в строке ("adimn" или "administr") пройдёт компилятор и превратит эндпоинт в недоступный никому. С енамом такой опечатки физически не может быть.

Стоимость фикса: 2-3 часа на всё API, делается одним PR.

Code smell #2: нет явного OWNER в 3 модулях — средний риск ›

Кто: fiscal, analytics, integrations.

В декораторах не указана роль owner. Owner всё равно проходит благодаря глобальному bypass'у в RolesGuard, но это создаёт «невидимую дверь» — в коде написано «только admin», в реальности и owner ходит.

Чем рискует: при аудите compliance трудно объяснить, почему в логах видно действие Owner на эндпоинте, который декоратор не разрешает. Также сложнее ответить на вопрос «кто может работать с фискалом».

Стоимость фикса: 30 минут — добавить "owner" в каждый @Roles этих модулей.

Code smell #3: бухгалтер закрыт от inventory — средний риск ›

Кто: модуль inventory, 14 эндпоинтов, у бухгалтера 0 доступа.

Это может быть правильно (склад — не дело бухгалтера), а может быть пропуском — например, при списании просрочки бухгалтеру нужно увидеть остатки.

Что делать: спросить вашего реального бухгалтера. Если ходит — добавить accountant хотя бы на read-методы.

Целевая модель — что мы решили

Финальный дизайн ролей, принятый 29 апреля 2026. Реализуется не перед пилотом, а через 4-6 недель после стабилизации.

Пять ролей в системе

Четыре staff-роли образуют иерархию делегирования. Client стоит отдельно — это покупатели с собственным кабинетом.

Staff — внутренняя команда

Owner — Jamshid
1 пользователь · все организации · создаёт Admin
↓
Admin — глава одной организации
создаёт Manager и Employee · обе мета-галочки
↓
Manager — руководитель команды
создаёт Employee только если Admin включил мета-галочку
↓
Employee — сотрудник
не создаёт никого · работает по галочкам
Шаблон «Наблюдатель»
Это Employee со всеми галочками только на чтение. Не отдельная роль — пресет в UI.

Client — покупатели

Client — клиент
сам регистрируется · свой кабинет
Параллельная ветка пользователей. Не часть staff-иерархии. Логин — через Telegram Mini App или phone+OTP. Видят только свой профиль, покупки, бонусы, ачивки, рефералов. Один клиент может покупать у нескольких франчайзи.

Жёсткие правила

Один Owner, всегда в плане

Owner — это Jamshid, единственный. В коде должна быть либо проверка по конкретному user_id, либо явное правило «owner — единственный». Никаких других owner'ов никогда.

Admin привязан к одной орг в плане

Не «доступ ко всем организациям» как сейчас. Один админ = одна организация. Для франчайзи-структуры с несколькими организациями — несколько разных админов.

Раздавать только своё частично есть

Каждый уровень может выдавать подчинённым только те права, которые есть у него самого. Попытка дать больше — отбивается на API уровне.

Мета-галочки создания в плане

Можно создавать менеджеров — отдельная галочка. Можно создавать сотрудников — отдельная. Admin имеет обе по дефолту, Manager получает «создавать сотрудников» только если Admin явно включил.

Изоляция Staff и Client в коде есть начало

Разные таблицы (users vs clients), разные эндпоинты логина (/auth/login vs /auth/client/telegram), разный audience в JWT (vendhub-users vs vendhub-clients). Клиентский токен физически не пройдёт на staff-эндпоинт.

Единая панель управления в плане

Всё управление пользователями — в одной странице, не разрозненными экранами «Менеджеры», «Операторы», «Бухгалтеры». Кнопка «Добавить» с выбором уровня и шаблона.

Современные фишки на потом — overkill для пилота

Эти возможности есть у крупных систем. Большинство overkill для 23 машин и команды до 20 человек. Помечено: есть/будет частично в очень далёкой очереди

Resource-level права решено по-другому

«Этот менеджер видит только эти 10 машин из 100». В целевой модели частично решается через привязку Admin к одной организации и через личные галочки прав.

Если нужна тонкая фильтрация по машинам внутри одной орг — добавится в Фазу 4.

Custom roles решено конструктором

Не нужны отдельные «кастомные роли» — конструктор галочек делает то же самое. Можно создать любую комбинацию прав без новой роли.

ABAC — контекст-зависимые правила далёкое будущее

Право зависит не только от галочки, но и от контекста: время суток, IP, устройство.

Пример: Operator может закрыть инкассацию только в рабочее время и только из мобильного приложения.

Risk-based 2FA далёкое будущее

2FA включается автоматом при подозрительном поведении: новый IP, новое устройство.

Approval workflows после первых месяцев

Чувствительные действия требуют второго подтверждения. Удаление пользователя, крупный перевод.

Зачем: когда команда подрастёт, важные действия не должны выполняться одним человеком.

Permission discovery UI после первых месяцев

Готовые отчёты «кто имеет доступ к X», «у кого права изменились за неделю».

Immutable audit storage после первых месяцев

Audit-логи в неизменяемом хранилище — даже Admin не может «подправить историю».

SCIM / автопровиженинг далёкое будущее

Синхронизация с HR-системой. Уволили в HR → отключили в VendHub.

Зачем: когда команда вырастет до 50+, ручная раздача станет узким местом.

Разрыв: было → станет

Что меняется в коде при переходе на целевую модель. Что остаётся, что переделывается, что добавляется.

Что остаётся без изменений

Сохраняется как есть

  • JWT-механизм с подписью, blacklist, refresh-rotation
  • Bcrypt для паролей, factor 12
  • 2FA с TOTP и backup-кодами
  • Lockout после 5 неудач на 30 минут
  • Throttle-лимиты на публичных эндпоинтах
  • Audit log на критические действия
  • RBAC canary тест в CI
  • Cookie с HttpOnly + Secure + SameSite
  • Telegram-вход с HMAC-проверкой
  • Локализация ролей в UI на ru/uz

Что переделывается

  • Енам UserRole сокращается с 7 до 4 значений
  • RolesGuard превращается в PermissionsGuard (или работают оба)
  • 1067 декораторов @Roles постепенно мигрируют на @Permissions
  • Создание пользователя — новая логика с проверкой «можешь дать?»
  • Добавляется таблица user_permissions
  • Добавляется таблица clients (миграция из users)
  • Admin теряет «доступ ко всем орг», получает «доступ к одной»
  • UI создания пользователей — новая страница-панель

Что не нужно делать в новой модели

Идея «custom roles» отпадает — конструктор галочек делает то же самое более гибко. «Resource-level права на машины» частично решается через привязку Admin к одной организации (если франчайзи = одна организация). «Field-level права» (бухгалтер видит зарплату, но не дату рождения) тоже отпадает на ближайшие годы — это enterprise-история.

Фаза 2 — первый месяц после запуска

Технический долг и улучшения, которые не блокируют пилот, но стоит закрыть в первые недели — пока система маленькая и менять её безболезненно.

Унифицировать стиль @Roles на енам 3 модуля сейчас MIXED, ~47 чисто string. Перевести всё на UserRole.X для типобезопасности — опечатка вроде "adimn" не пройдёт компилятор.
Явный owner в декораторах Добавить UserRole.OWNER в @Roles fiscal, analytics, integrations. Это для compliance и читаемости логов, не для безопасности (Owner и так проходит через bypass).
Typo guard для строковых ролей Тест, который проверяет: каждая строка в @Roles входит в UserRole-енам. Опечатка ловится в CI до деплоя.
Permission discovery — простой UI Страница в админке «список всех эндпоинтов и кто к ним имеет доступ». Парсится автоматически из @Roles при сборке. Полезно для compliance и для тебя как Owner.
Скрытие resource-level доступа в machine-access UI Если в Фазе 1 включили только сам механизм фильтрации, в Фазе 2 — добавить в UI визуальную индикацию «эта машина закреплена за этими X менеджерами», и быстрые кнопки массового назначения.

Фаза 3 — миграция модели ролей

Переход с 7 ролей на 4 staff + Client + конструктор галочек. Делается на стабилизированном пилоте, по мере появления потребности (второй админ, рост команды). С учётом готовности: фундамент уже в коде (RBAC модуль с динамическими ролями + permissions), остаётся доинтегрировать.

3.1 PermissionsGuard и интеграция с существующим RBAC Сделать новый PermissionsGuard, который читает permissions через RbacService (уже есть). Параллельно с @Roles ввести массовое использование @Permissions. Оба гарда работают одновременно — миграция постепенная.
3.2 Правило «нельзя выдать чего нет» В RbacService добавить assertCanGrant(grantorId, permission). Попытка дать роль/permission, которых нет у granter'а — отбивается. Это техническое усиление, которого сейчас нет.
3.3 Мета-галочки can_create_managers / can_create_employees Добавить две системные permissions в существующую таблицу permissions: users:create_manager и users:create_employee. Включаются индивидуально при создании Admin / Manager.
3.4 Сокращение enum UserRole с 7 до 4 Удаление OPERATOR, WAREHOUSE, ACCOUNTANT, VIEWER из основного enum. Они становятся пресетами галочек, не ролями. Скрипт миграции пересчитывает каждого существующего пользователя в нужный шаблон Employee. Owner — добавить уникальный constraint «единственный».
3.5 Admin привязан к одной организации Сейчас Admin может видеть все орг через Owner-bypass или прямо. Переделать так, чтобы Admin был жёстко связан с одной организацией через его user.organizationId. Только Owner видит все.
3.6 Разделение users и clients Новая таблица clients. Миграция всех клиентов из users. Отдельные эндпоинты /auth/client/telegram и /auth/client/phone-otp. Разный audience в JWT (vendhub-clients vs vendhub-users). Гарды отбивают чужой токен.
3.7 UI единой панели управления людьми Список всех пользователей орг с фильтрами. Кнопка «Добавить» (Manager / Employee, видна по мета-галочкам). Карточка пользователя с галочками 8 категорий + мета-галочки создания + закреплённые машины. История изменений прав. Шаблоны «Оператор», «Кладовщик», «Бухгалтер», «Наблюдатель» как кнопки-пресеты.
3.8 Миграция декораторов с @Roles на @Permissions 1067 декораторов перевести поэтапно. Можно по одному модулю за PR, оба декоратора параллельно работают. После полной миграции старые @Roles удаляются.

Фаза 4 — улучшения после миграции

Когда новая модель стабилизировалась — точечные улучшения, которые остались актуальны.

Approval workflows для крупных операций Удаление пользователя, перевод денег > 10М сум — требуют подтверждения второго пользователя.
Immutable audit log Перенести audit_log в append-only хранилище. Защита от того, что Admin подправит историю. Реализация через S3 с object lock или отдельную БД с DENY DELETE.
Risk-based 2FA 2FA автоматом включается при логине с нового IP/устройства. Снижает трение в обычной работе, поднимает защиту в риске.

Фаза 5 — enterprise-фичи на потом

Делать, только если попросит крупный клиент или команда вырастет до 50+ человек. Большинство — overkill для текущего масштаба.

SCIM провиженинг Автосинхронизация с HR-системами (1C, Bitrix24, Workday). Уволили в HR → отключили в VendHub.
JIT elevation + break-glass Временное повышение прав на задачу с автовозвратом. Экстренный доступ с автологом Owner'у.
ABAC с контекстом Правила «только в рабочее время + только из мобильного приложения». Реализация через Cedar / Casbin.
