# Telegram Bot Pilot Plan — VendHub OS

**Version:** 1.0  
**Date:** 2026-04-20  
**Sprint context:** Merged in Sprint D (2026-04-07), not yet battle-tested  
**Scope:** Staff Bot (operator-facing) only — Customer Bot pilot follows after Staff Bot passes

---

## 1. Цели пилота

| #   | Цель                                                       | Измерение                                               |
| --- | ---------------------------------------------------------- | ------------------------------------------------------- |
| 1   | Validate Staff Bot end-to-end with 1 real operator         | All 20 test scenarios executed                          |
| 2   | Collect real-world bugs and UX issues before wider rollout | Bug report count + severity breakdown                   |
| 3   | Measure bot adoption vs web panel                          | Tasks completed via bot / total tasks during pilot week |
| 4   | Make a data-driven rollout decision                        | Pass / Conditional / Fail verdict at end of Day 5       |

**Пилот касается только Staff Bot** (`TELEGRAM_BOT_TOKEN`).  
Сервисы в охвате: `BotHandlersService`, `BotMenuService`, `BotTaskOpsService`, `BotMachineOpsService`, `BotRouteOpsService`, `BotNotificationsService`, `BotStatsService`, `BotAdminService`.

---

## 2. Критерии успеха (PASS)

Все критерии должны быть выполнены одновременно.

- [ ] Оператор завершает **20+ задач через бота** за 5 дней (не через веб)
- [ ] **Zero data loss** — каждое изменение статуса задачи в боте отражается в базе данных (проверяется ежедневно через SQL-дамп)
- [ ] **Response time < 3 секунд** для команд `/start`, `/tasks`, `/machine`, `/route` (измеряется логами API)
- [ ] На выходном интервью оператор отвечает: **"Я бы использовал это ежедневно"** (или эквивалент на русском/узбекском)
- [ ] **Менее 5% команд завершаются ошибкой** (error rate по Sentry за 5 дней)

---

## 3. Критерии провала / Abort

Если **любой** из этих критериев сработает — пилот немедленно останавливается.

| Критерий                                         | Порог                                                                               | Действие                       |
| ------------------------------------------------ | ----------------------------------------------------------------------------------- | ------------------------------ |
| Потеря данных                                    | Любой случай: задача отмечена выполненной в боте, но не в БД (или наоборот)         | СТОП. Hotfix до возобновления  |
| Высокий error rate                               | >10% команд за любой день                                                           | СТОП. Диагностика, переоценка  |
| Оператор называет бота "сломанным и непригодным" | Любое время в течение пилота                                                        | СТОП. Структурированный разбор |
| Сбой авторизации                                 | Незарегистрированные Telegram ID получают доступ к данным (не сообщение "отказано") | СТОП немедленно                |

## 4. Test Scenarios — 20 сценариев

### Легенда статусов

- [ ] — не протестировано
- [x] — пройдено
- [!] — упало (прикрепить баг-репорт)

### A. Onboarding (5 сценариев)

**День 1 — обязательно все 5 пройти до начала реальной работы.**

- [ ] **A1. Первый запуск**
  - Оператор набирает `/start` в чате с ботом
  - Ожидаемое: приветственное сообщение на русском языке с именем оператора и инструкцией
  - Проверить: сообщение не на английском, нет `undefined`, нет JSON-мусора

- [ ] **A2. Активация по ссылке**
  - Менеджер отправляет оператору deep-link вида `t.me/VendHubStaffBot?start=<token>`
  - Оператор кликает по ссылке, бот инициализирует сессию автоматически
  - Ожидаемое: сессия создана, оператор видит своё меню без ручной настройки

- [ ] **A3. Корректные данные профиля**
  - После `/start` или `/profile` оператор видит: имя, роль (operator/manager), назначенный маршрут
  - Ожидаемое: все поля заполнены и совпадают с записью в базе данных

- [ ] **A4. Отказ незарегистрированному пользователю**
  - Тестировщик пишет `/start` с Telegram ID, которого нет в системе
  - Ожидаемое: вежливое сообщение "Доступ запрещён, обратитесь к администратору"
  - Проверить: **нет краша**, нет утечки данных о системе, Sentry не фиксирует unhandled exception

- [ ] **A5. Переключение организации**
  - Оператор, привязанный к 2+ организациям, видит меню выбора организации при старте
  - После выбора все последующие команды выполняются в контексте выбранной организации
  - Проверить: задачи другой организации не видны

---

### B. Task Management (5 сценариев)

**Дни 2-3 — основная рабочая нагрузка.**

- [ ] **B1. Список задач с приоритетом**
  - Команда `/tasks` или кнопка "Мои задачи"
  - Ожидаемое: список задач, отсортированных по приоритету (HIGH → MEDIUM → LOW), потом по дедлайну
  - Проверить: задачи других операторов не отображаются

- [ ] **B2. Детали задачи**
  - Оператор открывает задачу из списка
  - Ожидаемое: отображаются все поля — машина (ID + локация), продукт, количество, заметки, дедлайн, статус
  - Проверить: нет пустых полей без причины, фото из прошлых выполнений доступно

- [ ] **B3. Принятие задачи**
  - Оператор нажимает "Принять задачу" на задаче со статусом `pending`
  - Ожидаемое: статус меняется на `in_progress` в боте и в базе данных
  - SQL-проверка: `SELECT status FROM tasks WHERE id = <id>` — должно быть `in_progress`
  - Проверить: другой оператор больше не видит кнопку "Принять" для этой задачи

- [ ] **B4. Завершение задачи с фото**
  - Оператор фотографирует результат, отправляет фото в чат задачи
  - Ожидаемое: статус меняется на `done`, фото прикреплено к задаче, менеджер получает уведомление
  - SQL-проверка: поле `completionPhotoUrl` в tasks не null
  - Проверить: фото доступно в веб-интерфейсе

- [ ] **B5. Защита от чужих задач**
  - Оператор A пытается принять задачу, назначенную Оператору B
  - Ожидаемое: сообщение "Эта задача назначена другому исполнителю"
  - Проверить: статус задачи не изменился в БД

---

### C. Machine Operations (4 сценария)

**Дни 2-3 — параллельно с Task Management.**

- [ ] **C1. Запрос статуса машины**
  - Оператор отправляет `/machine VM-01` (или использует меню поиска)
  - Ожидаемое: карточка машины — название, локация, статус (online/offline/maintenance), последнее обслуживание
  - Проверить: нет задержки >3 секунд

- [ ] **C2. Текущие запасы и алерты**
  - Из карточки машины оператор переходит в раздел запасов
  - Ожидаемое: список слотов с текущим количеством, подсвечены позиции ниже минимального порога
  - Отдельно: отображается дата последнего пополнения для каждой позиции
  - Проверить: данные совпадают с веб-панелью (spot-check 3-5 позиций)

- [ ] **C3. Ручное обновление остатков**
  - Оператор обновляет количество для позиции без проведения транзакции
  - Ожидаемое: поле `currentQuantity` в inventory обновляется, создаётся запись в audit log
  - SQL-проверка: `SELECT updated_at FROM inventory WHERE slot_id = <id>`

- [ ] **C4. Сообщение об инциденте с фото**
  - Оператор использует кнопку "Сообщить о проблеме" на карточке машины
  - Вводит описание, отправляет фото
  - Ожидаемое: создаётся инцидент в системе, менеджер получает уведомление, фото прикреплено
  - Проверить: инцидент виден в веб-интерфейсе с правильным machine_id

---

### D. Route Tracking (4 сценария)

**День 4 — с реальным маршрутом.**

- [ ] **D1. Уведомление о начале маршрута**
  - В 09:00 оператор получает push-уведомление с деталями маршрута на день
  - Ожидаемое: список остановок, оценочное время, первая остановка выделена
  - Проверить: уведомление доставлено (Telegram delivery receipt), время соответствует расписанию

- [ ] **D2. Начало маршрута**
  - Оператор нажимает "Начать маршрут"
  - Ожидаемое: статус маршрута меняется на `in_progress`, бот просит подтвердить GPS-доступ (или принять первую остановку)
  - SQL-проверка: `SELECT started_at FROM routes WHERE id = <id>`

- [ ] **D3. Пошаговые инструкции по остановкам**
  - Оператор отмечает первую остановку выполненной
  - Ожидаемое: бот автоматически показывает следующую остановку с адресом и задачами
  - Проверить: порядок остановок соответствует оптимизированному маршруту из БД

- [ ] **D4. Завершение маршрута**
  - Оператор нажимает "Завершить маршрут" после последней остановки
  - Ожидаемое: сводка маршрута (N остановок, время в пути, выполненные задачи) отправляется оператору
  - Менеджер получает аналогичную сводку
  - SQL-проверка: `SELECT completed_at FROM routes WHERE id = <id>` не null

---

### E. Notifications (2 сценария)

**День 5 — тестирование с искусственно вызванными условиями.**

- [ ] **E1. Алерт о просроченной задаче**
  - Создать тестовую задачу с дедлайном 2+ часа назад, не менять статус
  - Ожидаемое: оператор получает уведомление "Задача просрочена: <название>, дедлайн был <время>"
  - Проверить: уведомление содержит кнопки быстрых действий (Принять / Открыть)

- [ ] **E2. Алерт о предсказанном дефиците (REFILL_NOW)**
  - Система `predictive-refill` генерирует рекомендацию `REFILL_NOW` для тестовой машины
  - Ожидаемое: оператор получает уведомление "⚠️ Машина VM-XX: ожидается дефицит через <N> часов. Создать задачу на пополнение?"
  - Уведомление содержит кнопку действия — нажатие создаёт задачу немедленно
  - Проверить: задача создана в БД с правильным machine_id и product_id

## 5. Шаблон баг-репорта

Оператор отправляет репорт **напрямую менеджеру в Telegram** в следующем формате:

```
БАGREPORT #<порядковый_номер>

Дата/время: <когда произошло>
Сценарий: <например, B4 — завершение задачи с фото>

Что я пытался сделать:
<1-2 предложения>

Что произошло:
<конкретно: сообщение об ошибке, зависание, неверные данные>

Скриншот/видео:
<приложить к этому сообщению>

Серьёзность:
[ ] CRITICAL — данные повреждены, работа невозможна
[ ] HIGH — важная функция не работает, обходной путь есть
[ ] MEDIUM — неудобство, но задачу можно выполнить
[ ] LOW — косметический баг, опечатка

Обходной путь (если есть):
<что помогло продолжить работу>
```

**Дополнительно:** менеджер ведёт [лог багов](../bugs/telegram-bot-pilot-bugs.csv) с полями: `id, date, scenario, severity, status, assignee, fixed_in`.

## 6. Протокол ежедневного check-in

### Оператор (каждый вечер до 18:00)

Отправляет менеджеру в Telegram:

```
ДЕНЬ <N> ИТОГ

Задачи через бот: <количество>
Задачи через веб: <количество>
Баги сегодня: <количество и ссылки на BUGREPORT-ы>
Общее впечатление: <1-2 предложения>
Блокеры на завтра: <есть / нет, если есть — описать>
```

### Менеджер (каждый вечер)

- [ ] Получить check-in от оператора
- [ ] Просмотреть новые баг-репорты
- [ ] Назначить серьёзность и приоритет
- [ ] CRITICAL/HIGH баги — открыть hotfix-ветку немедленно (`hotfix/tg-bot-pilot-day<N>`)
- [ ] Обновить [лог багов](../bugs/telegram-bot-pilot-bugs.csv)
- [ ] Сообщить оператору о прогрессе по CRITICAL/HIGH (до 20:00)

### CRITICAL баг — протокол

1. Менеджер получает баг-репорт с отметкой CRITICAL
2. Пилот **приостанавливается** до исправления
3. Фикс деплоится в тот же день
4. Тест-кейс повторяется перед возобновлением
5. Если фикс не готов за 24 часа — оценить Abort-критерий

## 7. Timeline пилота (5 дней)

| День            | Фокус                                 | Сценарии              | Ожидаемый результат                     |
| --------------- | ------------------------------------- | --------------------- | --------------------------------------- |
| **День 1** (Пн) | Onboarding + установка                | A1–A5                 | Все 5 пройдены, оператор готов к работе |
| **День 2** (Вт) | Задачи + первое знакомство с машинами | B1–B5, C1–C2          | Оператор управляет задачами через бот   |

## 8. Выходное интервью (День 5)

Менеджер проводит интервью голосом или текстом в Telegram (~20 минут).

### Вопросы

1. **Ежедневное использование:** "Ты бы использовал этого бота каждый день, если бы он уже был готов к работе? Почему да / почему нет?"

2. **Топ-3 проблемы:** "Назови три самые раздражающие вещи, с которыми ты столкнулся. Что мешало работе больше всего?"

3. **Топ-3 удачных момента:** "Что работало хорошо и тебе понравилось? Что ты хотел бы видеть в боте всегда?"

4. **Сравнение с текущим процессом:** "По сравнению с тем, как ты работаешь сейчас (веб/бумага/звонки) — бот работает: быстрее / примерно так же / медленнее? Конкретные примеры?"

5. **Рекомендация коллегам:** "Ты бы порекомендовал другим операторам перейти на бота? Что бы ты им сказал?"

6. **Ключевая фича:** "Если бы ты мог добавить одну функцию, которой сейчас нет — что это было бы?"

7. **Откат:** "Есть ли что-то в боте, что хуже текущего процесса настолько, что ты предпочёл бы вернуться к старому?"

### Запись интервью

- [ ] Ответы записаны (текстом или аудио)
- [ ] Ключевые цитаты выделены для отчёта
- [ ] Ответ на вопрос 1 зафиксирован отдельно — он входит в PASS-критерий

## 9. Дерево решений post-pilot

```
Пилот завершён → Подсчёт метрик
        │
        ├── Все PASS-критерии выполнены?
        │       │
        │       YES ──────────────────────────────► PASS
        │       │                                   Развернуть на 3-5 операторов
        │       │                                   в течение 1 недели
        │       │                                   Открыть Customer Bot pilot
        │       │
        │       NO
        │       │
        ├── Abort-критерий сработал?
        │       │
        │       YES ─────────────────────────────► FAIL
        │       │                                  Задокументировать всё
        │       │                                  Приостановить бот-роллаут
        │       │                                  Сфокусироваться на веб-приложении
        │       │                                  Запланировать пересмотр через 4 недели
        │       │
        │       NO
        │       │
        └── 1-2 PASS-критерия не выполнены?
                │
                YES ─────────────────────────────► CONDITIONAL
                                                   Исправить топ-5 багов
                                                   Повторный пилот 3 дня
                                                   с тем же оператором
                                                   Затем повторная оценка:
                                                   PASS или FAIL (не CONDITIONAL снова)
```

### Сроки по каждому вердикту

| Вердикт     | Следующий шаг                                      | Срок     |
| ----------- | -------------------------------------------------- | -------- |
| PASS        | Onboarding 3-5 операторов, мониторинг ещё 1 неделю | +7 дней  |
| CONDITIONAL | Исправление багов + повторный пилот                | +14 дней |
| FAIL        | Отчёт с находками, баг-беклог заморожен            | +3 дня   |

## 10. Технические предварительные условия

Все пункты должны быть отмечены до Дня 1.

### Конфигурация

- [ ] `TELEGRAM_BOT_TOKEN` настроен в production environment (не dev токен)
- [ ] `DISABLE_BOT_POLLING=false` — polling активен (или webhook настроен корректно)
- [ ] `TELEGRAM_CUSTOMER_BOT_TOKEN` отключён для периода пилота (чтобы не создавать путаницу)
- [ ] Переменная `BOT_PILOT_MODE=true` добавлена (если такой флаг существует для verbose-логирования)

### База данных

- [ ] Telegram ID оператора-пилота добавлен в запись `User` в БД
- [ ] Роль оператора корректно настроена (`operator`, не `admin` и не `customer`)
- [ ] Тестовые машины (`VM-01`, `VM-02`) существуют с реальными данными слотов
- [ ] Тестовый маршрут создан и назначен оператору для Дня 4

### Мониторинг

- [ ] Sentry настроен и принимает события от bot-сервисов
- [ ] Алерт в Sentry → канал менеджера при `error.level = fatal` или `critical`
- [ ] API-логи доступны (Railway / локальный сервер) для проверки response time
- [ ] Ежедневный SQL-дамп задач настроен для data loss проверки

### На стороне оператора

- [ ] Оператор установил Telegram (актуальная версия, iOS или Android)
- [ ] Оператор знает, как отправить фото в чат
- [ ] Оператор знает, как отправить голосовое / видео-сообщение (для тестирования Edge cases)
- [ ] Оператор получил контакт менеджера для экстренной связи
- [ ] Оператор ознакомлен с шаблоном баг-репорта (см. Раздел 5)
- [ ] Оператор знает: если бот не отвечает >10 секунд — сообщить немедленно

### Резервный план (если бот упал)

- [ ] Оператор может переключиться на веб-интерфейс без потери данных
- [ ] Менеджер знает команду для быстрого перезапуска bot-polling: `pm2 restart api` / Railway redeploy
- [ ] Hotfix-ветка может быть задеплоена за < 30 минут
