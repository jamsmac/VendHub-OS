# UI Audit Report: VendHub OS

Date: 2026-03-12  
Scope: `apps/web`, `apps/client`, `apps/mobile`  
Excluded from primary scope: `apps/site` runtime review

## Executive Summary

Полный UI-аудит подтвердил, что основная интерфейсная поверхность существует и частично работает, но production-ready состояние по UI и client-side integration не достигнуто.

Наиболее критичные проблемы сосредоточены в `apps/web` и `apps/mobile`:

- в `web` сломаны или нестабильны несколько business-critical страниц (`orders`, `payments`, `reports`, `webhooks`);
- logout в shell админки является визуальным элементом без реального завершения сессии;
- в `client` локальная интеграция блокируется собственной CSP, а `QR`-сценарий падает в runtime;
- `mobile` не удалось поднять вообще: `expo start` падает на отсутствующем `async-limiter`, поэтому runtime-проверка мобильного приложения заблокирована;
- модель хранения токенов и кэширования в клиентах расходится с заявленной и расширяет риск stale/sensitive data exposure.

Итог по UI readiness: `no-go` для уверенного production release без исправления top-priority дефектов ниже.

## Verification Boundary

Runtime-проверка выполнена:

- `apps/web` на `http://localhost:3000`
- `apps/client` на `http://localhost:5174`
- backend health через `http://localhost:4000/api/v1/health`

Runtime-проверка не выполнена:

- `apps/mobile`
  - причина: `pnpm --filter @vendhub/mobile start` падает с `Cannot find module 'async-limiter'`

Допущения методики:

- успешный login flow в `web` не был полностью пройден через UI из-за backend rate limit после невалидных попыток;
- для проверки protected dashboard routes использовался локально сгенерированный JWT на основе seeded user `admin@vendhub.uz`, чтобы проверить shell, redirects и реальные экраны без изменения кода приложения;
- для `mobile` вывод ограничен static audit + navigation/auth/store/API audit.

## Discovery and Interface Map

### Web Admin

- Framework surface: Next.js App Router
- Auth/public entry: `/auth`, `/auth/register`, `/auth/reset-password`
- Protected shell: `dashboard/layout.tsx` + `Sidebar` + `Header`
- Auth/session layer:
  - zustand persist store
  - axios client with refresh flow
  - access token хранится в `localStorage`
- i18n layer:
  - `next-intl`
  - runtime найдены missing message keys
- Notable route shape:
  - полноценный `/dashboard/*`
  - параллельно существуют top-level aliases `/machines`, `/reports`, `/tasks`, `/users` и т.д.

### Client PWA

- Framework surface: Vite + React + React Router
- Shell: `Layout`
- Public entry: `/`
- Protected routes обёрнуты в `ProtectedRoute`
- Auth/session layer:
  - zustand
  - axios with refresh
  - access token хранится в `localStorage`
- Telegram-specific integration:
  - `useTelegramAuth`
  - вне Telegram flow фактически no-op
- Offline/cache layer:
  - PWA service worker
  - runtime caching для API домена
- i18n layer:
  - `react-i18next`

### Mobile

- Framework surface: React Native + Expo
- Root branching:
  - `RootNavigator` -> `AuthNavigator` / `MainNavigator` / `ClientNavigator`
- Session layer:
  - `SecureStore` для токенов
- Offline/cache layer:
  - `AsyncStorage` persister для React Query cache
  - `offlineFirst`
- i18n layer:
  - `react-i18next`

### Site Anomaly

`site` не отсутствует, но живёт не по ожидаемому пути `src/app`, а по пути `apps/site/app`.  
Подтверждённая UI-поверхность:

- `apps/site/app/[locale]`
- `apps/site/app/[locale]/admin/*`
- `apps/site/app/[locale]/machines/*`

Вывод:

- `site` не включён в основной UI-аудит по исходному scope;
- path layout `app/` вместо `src/app/` является source of confusion для tooling/discovery и должен быть отражён в документации и внутренних соглашениях.

## Findings

### UI-001

- Severity: `High`
- App: `web`
- Category: `Security | State Handling`
- Screen / Route: dashboard shell header, avatar dropdown
- Evidence:
  - `apps/web/src/components/layout/header.tsx:122-131`
  - dropdown renders menu items only; logout item has no handler
  - runtime: click `Выйти` on `/dashboard/machines`
  - localStorage still contains `vendhub_access_token` and `vendhub-auth`
  - route remains on dashboard
- Reproduction path:
  1. Open any protected dashboard route with valid session
  2. Open avatar dropdown
  3. Click `Выйти`
  4. Observe no redirect, no token cleanup, no state reset
- Why it matters:
  - пользователь получает ложное ощущение выхода из системы;
  - на shared device access сохраняется;
  - session hygiene ломается именно в shell, где пользователь ожидает безопасный logout.
- Recommendation:
  - wire logout item to auth store logout + token cleanup + redirect to `/auth`;
  - добавить e2e smoke test на logout в desktop и mobile header.
- Confidence: `High`

## UI Debt Patterns

### 1. Contract Drift Between API Wrappers and Screens

Several pages still make their own assumptions about envelope shape (`res.data`, `res.data.data`, arrays, pagination objects) even though a global response interceptor already mutates the response. Это root cause сразу нескольких админских поломок.

### 2. Security Model Drift

В комментариях и ожиданиях декларируется one model (`in-memory only`), а в коде реализована другая (`localStorage` / PWA cache / broad offline persistence). Это создаёт ложную уверенность и повышает шанс пропустить реальные session/data risks.

### 3. Incomplete Shell Hardening

Ключевые shell-элементы, вроде logout и header i18n, недостаточно защищены smoke-проверками. В результате ломаются не edge cases, а базовые элементы навигации и сессии.

### 4. Testability Gap

`client` локально режется CSP, `mobile` не стартует, а `web` требует обхода throttling для продолжения audit. Это означает, что UI regression loop слишком хрупкий даже до уровня E2E.

### 5. i18n as Afterthought

Несколько пользовательских surface’ов показывают raw keys или undefined-derived labels. Это указывает на отсутствие обязательных i18n assertions в shared shell и critical tables.

## Final Verdict

Можно ли считать текущий UI слой production-ready: `no`

Причины:

- `web` теряет устойчивость именно на critical admin surfaces;
- `client` имеет слабый auth recovery UX и broken QR flow;
- `mobile` не проходит даже базовый runtime boot;
- client-side session/caching behavior нуждается в жёстком выравнивании с security model.

Минимальный gate перед следующим release-кандидатом:

1. Починить `web` logout.
2. Вернуть в рабочее состояние `orders`, `payments`, `reports`, `webhooks`.
3. Починить `client:/scan`.
4. Устранить CSP block для local QA.
5. Восстановить `mobile` startup и выровнять auth recovery contract.
